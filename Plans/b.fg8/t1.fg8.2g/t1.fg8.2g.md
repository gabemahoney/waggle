## Description

Strip Waggle's last persistent-state and daemon-era surfaces, leaving only stdio MCP + tmux + Claude Status integration.

Persistence removal (SR-9.1):
- Delete `src/waggle/schema.sql` and `src/waggle/database.py`.
- Remove every `import sqlite3` from `src/waggle/` (including any indirect importers). Waggle source must not import any SQLite driver in normal operation.
- Delete `tests/test_database.py` and the `db_path` and `registered_caller` fixtures (SR-11.3) wherever they live.

Tmux state-detection cleanup (SR-6.2):
- Delete `src/waggle/state_parser.py` and `tests/test_state_parser.py`.
- Delete the pane-snapshot fixtures under `tests/fixtures/pane_snapshots/` (SR-11.3).
- Remove `tmux.is_llm_running` and any other state-classification helpers from `tmux.py`. State classification is Claude Status's responsibility now.

Config cleanup (SR-8.1):
- Reduce `src/waggle/config.py` to env-var helpers (e.g. an override for the tmux executable path) or delete it outright if nothing meaningful remains. The MCP server must start and operate with no `~/.waggle/config.json` file present.
- Remove all daemon-era keys: `db_path`, `queue_path`, `http_port`, `relay_timeout_seconds`, `max_workers`, repository paths, CMA endpoints, admin emails. The `get_output` scrollback default is hardcoded (50) per Epic 3 / SR-3.4 — not an env-var knob.
- Update `tests/test_config.py` to match (or delete if config is gone).

Documentation finalization (SR-12):
- Add `docs/migration.md` describing the hard cutover (no in-place upgrade path): stop and disable the old `waggle.service`, uninstall the old daemon, delete the old SQLite database file, install the new package, run `waggle install`, restart any orchestrator stdio sessions.
- Final pass on `docs/architecture/`: keep only material that describes the stdio MCP launch model, the six tools, and the tmux + Claude Status integration. Remove or replace any remaining daemon-era diagrams.
- Final README pass: ensure the migration doc is linked from the README and the surviving-tools list matches the actual MCP surface exactly.

## Acceptance Criteria

- `grep -REn 'sqlite3|sqlalchemy' src/waggle/` returns nothing.
- `find src/waggle/ -name schema.sql -o -name database.py -o -name state_parser.py` returns nothing.
- `find tests/fixtures/pane_snapshots` returns nothing.
- `pytest -q` is green; the test count plausibly reflects the simplified surface (no daemon, no DB, no state parser, six MCP tools).
- The user can run `waggle mcp` on a host with no `~/.waggle/` directory at all (verifiable with `rm -rf ~/.waggle` first) and the server starts and serves the six tools.
- On a fresh VM following only the README + `docs/migration.md`, the user can: install Claude Status, install Waggle, run `waggle install`, run `waggle mcp`, spawn a worker, send input, get output, answer a question, and terminate the worker — all end-to-end.
- `docs/migration.md` exists and walks an operator from a working old-Waggle host (with daemon + SQLite) to a working new-Waggle host (stdio + Claude Status) without ambiguity.
