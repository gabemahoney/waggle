## Context

Per SR-12, write `docs/migration.md` describing the hard cutover from old daemon-era Waggle to stateless stdio Waggle. Final pass on `docs/architecture/`.

## What Needs to Change

**Write `docs/migration.md`:**
- Hard cutover, no in-place upgrade.
- Operator steps:
  1. Stop and disable the old `waggle.service` (if installed).
  2. Uninstall the old `waggle` package.
  3. Delete the old SQLite database file (default path or operator-known location).
  4. Install the new `waggle` package (now stdio-only).
  5. Run `waggle install` (which invokes `claude-status install-hooks`).
  6. Restart any orchestrator stdio sessions; they pick up the new `waggle mcp` server.
- Note that orchestrators that previously called `register_caller`, `list_workers`, `check_status`, `approve_permission` must be updated to call `claude-status workers` / `worker` / `decide` directly.

**Final pass on `docs/architecture/`:**
- Keep only material describing the stdio MCP launch model, the six tools, and the tmux + Claude Status integration.
- Remove or replace any remaining daemon-era diagrams that survived Epic 6's `docs/architecture/` cleanup.

**Link from README:**
- Add a "Migration" section in `README.md` linking to `docs/migration.md`.

## Constraints

- Migration doc is single-page; brief and operationally precise.
- No in-place upgrade path. Hard cutover.

## Acceptance Criteria

- `docs/migration.md` exists with at minimum the six operator steps in order.
- README's Migration section links to `docs/migration.md`.
- An operator can follow the README + migration doc on a fresh VM and end up with a working `waggle mcp` end-to-end.
- `docs/architecture/` has zero daemon-era diagrams.
