## Description

Add `src/waggle/claude_status.py` — a subprocess-based client for the documented Claude Status consumer-CLI verbs (`workers`, `worker`, `capabilities`). Each invocation is a fresh subprocess (SR-2.3): JSON-parse stdout on exit 0; on non-zero, parse the stderr envelope grammar `ERROR: <ErrName>: <description>
` and surface `ErrName` verbatim. Pin the consumer contract: at startup (and lazily as needed) call `claude-status capabilities` and refuse to continue if `contract_version` major != `1` (SR-2.2). Map all client outcomes into the SR-7.1 structured operation-failed shape so callers in later Epics can wrap tool bodies uniformly. Add the shared test infrastructure from SR-11.2: `tests/sample_payloads.py` (canonical Worker JSON, pending-payload JSON, stderr envelope strings), and helpers `fake_claude_status`, `fake_worker_record(instance_id, status, labels=None, pending=None)`, `fake_workers_response(records, skipped=())`, `fake_tmux_pane(text)`. Unit tests in `tests/test_claude_status_client.py` exercise: `workers` with and without `--label` filter; `worker <id>`; capabilities pin (accepts 1.x, refuses 2.x); each typed error (`ErrInstanceNotFound`, `ErrNoPendingRequest`, `ErrSchemaMismatch`, `ErrStoreUnavailable`, `ErrPayloadMalformed`) maps to operation-failed without retry. No removals from existing code in this Epic.

## Acceptance Criteria

- All new unit tests in `tests/test_claude_status_client.py` pass under the existing `pytest -q` invocation.
- Existing Waggle test suite remains green (no regressions).
- The user can run `python -c 'from waggle.claude_status import workers; import json; print(json.dumps(workers(label="waggle_owned=1")))'` and observe a parsed `{"workers": [...], "skipped": [...]}` envelope.
- The user can stop the Claude Status DB (e.g. `rm $CLAUDE_STATUS_DB`) and re-run the above; the client returns an operation-failed result whose `err_name` is `ErrStoreUnavailable`, and the Python process exits 0 (no traceback).
- The user can shadow `claude-status` with a stub that prints a `capabilities` JSON whose `contract_version` is `"2.0.0"`; the client's startup check refuses with operation-failed and an `err_name` indicating contract-version mismatch.
