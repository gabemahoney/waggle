## Context

Tests across this Epic and sibling Epics 2–5 reference canonical Worker JSON, pending-payload JSON, capabilities payloads, and stderr envelope strings. Inlining these literals across test files causes drift and duplicates the contract surface. SR-11.2 calls for a single canonical constants module under `tests/`. This Subtask delivers it.

## What Needs to Change

Create `tests/sample_payloads.py` exporting:

- Canonical Worker JSON dicts conforming to Claude Status contract `1.0.0` — one per distinct shape tests will assert on (e.g. a working worker, an `ask_user` worker with single-question pending, an `ask_user` worker with multi-question pending, an `ended` worker, a worker with malformed labels surfaced in the `skipped` array). Do not ship six near-identical full-population dicts; trim to the variants the test suite actually exercises.
- Canonical pending-payload dicts: a `permission_request` payload with `kind`, `request_id`, `tool_name`, `tool_input`; an `ask_user_question` payload with a single-question `tool_input.questions` list; an `ask_user_question` payload with a two-question variant for the SR-3.5 multi-question refusal test in Epic 4.
- Capabilities response dicts: one with `contract_version="1.0.0"` (accepted) and one with `contract_version="2.0.0"` (refused). Both per the Claude Status `capabilities` verb shape.
- Stderr envelope strings for each documented `ErrName`: `ErrInstanceNotFound`, `ErrNoPendingRequest`, `ErrSchemaMismatch`, `ErrStoreUnavailable`, `ErrPayloadMalformed`. Each string conforms to the grammar `ERROR: <ErrName>: <description>
` exactly.

The module contains constants only — no logic, no side effects, no import-time computation that calls external systems.

## Key Files

- `tests/sample_payloads.py` — new file

## Acceptance Criteria

- `from tests.sample_payloads import <name>` succeeds for every public constant defined in the module; no side effects at import time.
- Each defined stderr envelope string starts with `"ERROR: "`, contains the documented `ErrName` substring verbatim, and ends with `"
"`.
- The capabilities response dicts contain a `contract_version` key whose value is a string.
- The Worker JSON dicts contain at minimum `instance_id`, `status`, `labels`, and `pending` keys; `status` is one of the six documented values; `labels` is a dict; `pending` is either `None` or a dict with a `kind` key.
- No import of `jsonschema` or any other validation library. No imports from `waggle` source.
