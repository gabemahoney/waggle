## Context

The user-facing entry point for the new stdio server is `waggle mcp`. SR-4.1 lists `waggle mcp` as a new subcommand. SR-2.2 requires the consumer-contract capability pin to fire at MCP server startup, NOT at module import — this Subtask is where the pin lands.

## What Needs to Change

- In `src/waggle/cli.py`, register a new `mcp` subparser under the existing subparsers construct. No flags on the subparser (no `--port`, `--host`, `--bind`, or any other transport-binding argument).
- The `mcp` subcommand handler:
  - Calls `waggle.claude_status.capabilities()` BEFORE serving stdio.
  - If operation-success and `contract_version` major == 1, proceeds to invoke the stdio-server module's `run()` entry point.
  - If operation-failed (any reason — missing binary, contract mismatch, unparseable, store unavailable), writes the structured error to stderr in the SR-7.1 envelope shape and exits non-zero. Does NOT start serving stdio.
- The `mcp` subcommand exits cleanly (exit 0) on stdin EOF.

## Constraints

- Do not modify the existing `serve`, `set-state`, `permission-request`, `ask-relay`, or `sting` subcommands.
- No new imports of `starlette`, `uvicorn`, or socket-binding modules in `cli.py`.

## Key Files

- `src/waggle/cli.py` — add the `mcp` subparser and dispatch

## Acceptance Criteria

- `waggle mcp --help` exits 0 and prints help text describing the stdio MCP server. No `--port`/`--host`/`--bind` options listed.
- On a host where `claude-status` is missing from PATH, `waggle mcp` exits non-zero with the SR-7.1-shaped structured error on stderr; never invokes the stdio server's `run()`.
- On a host where `claude-status capabilities` returns `contract_version="2.0.0"`, `waggle mcp` exits non-zero with a contract-mismatch err_name on stderr; never invokes `run()`.
- On a green host, `waggle mcp` invokes the stdio-server `run()` (verifiable by patching that entry point in a test).
- On stdin EOF, the subcommand exits 0 with no leaked subprocesses or threads.
- `waggle serve`, `waggle set-state`, `waggle permission-request`, `waggle ask-relay`, `waggle sting` continue to dispatch to their existing handlers unchanged.
