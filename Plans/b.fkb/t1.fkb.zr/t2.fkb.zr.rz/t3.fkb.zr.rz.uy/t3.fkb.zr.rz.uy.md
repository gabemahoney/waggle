## Context
After Epic 2, the package is at `src/claude_spawn/` and `claude_spawn.spawn` is the module that spawns tmux workers, lists them via Claude Status, and answers AUQ questions on their behalf. The four worker label env vars Claude Spawn injects, and the matching label-key reads used to find workers back, still carry the `WAGGLE`/`waggle_` namespace. This subtask flips all of them in one file, in lockstep, so the spawn/list/answer paths remain consistent end-to-end.

Tuple order in `_REQUIRED_ENV_VARS` is preserved per SR-4.2 — tests iterate the tuple by index and would otherwise break.

## What Needs to Change
- `_REQUIRED_ENV_VARS` tuple — rename the four label env-var names from `CLAUDE_STATUS_LABEL_WAGGLE_OWNED`, `CLAUDE_STATUS_LABEL_WAGGLE_SESSION_NAME`, `CLAUDE_STATUS_LABEL_WAGGLE_MODEL`, `CLAUDE_STATUS_LABEL_WAGGLE_REPO` to `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_OWNED`, `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_SESSION_NAME`, `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_MODEL`, `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_REPO`. Tuple order is preserved: the three non-label env vars (`CLAUDE_STATUS_INSTANCE_ID`, `CLAUDE_STATUS_RELAY_MODE`, `CLAUDE_STATUS_AUQ_MODE`) stay first, the four renamed label env vars follow in OWNED → SESSION_NAME → MODEL → REPO order.
- `spawn_worker_impl` — update the four label-bearing `(key, value)` pairs in the env-injection list that flows into `tmux new-session -e KEY=val` so the env-var keys match the new tuple. Values are unchanged. The three non-label entries are unchanged.
- `list_spawned_workers_impl` — change the `claude_status.workers(label=...)` filter argument from `"waggle_owned=1"` to `"claude_spawn_owned=1"`. Change the `labels.get("waggle_session_name", "")` projection read to `labels.get("claude_spawn_session_name", "")`.
- `answer_question_impl` — change the `claude_status.workers(label=...)` filter argument from `"waggle_owned=1"` to `"claude_spawn_owned=1"`. Change the `(matched.get("labels") or {}).get("waggle_session_name", "")` read to `(matched.get("labels") or {}).get("claude_spawn_session_name", "")`.

## Out of scope (deferred — DO NOT TOUCH)

- `FastMCP("waggle-stdio")` in `mcp_stdio.py` → Epic 5.
- Module docstring lines that mention "waggle.spawn._tmux" patch target, "Fetch waggle-owned workers", or the `"waggle list_spawned_workers: skipped row..."` stderr log prefix → Epic 7 prose sweep.
- The `from waggle import claude_status` line (still reads `waggle` because the live tree hasn't had Epic 2 applied yet, depending on commit order).

## Key Files

## Acceptance Criteria
- `_REQUIRED_ENV_VARS` is a 7-tuple. First three entries unchanged. Last four entries are `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_OWNED`, `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_SESSION_NAME`, `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_MODEL`, `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_REPO` in that order.
- After `spawn_worker_impl` runs, the argv handed to `_tmux(["new-session", ...])` contains exactly four `-e CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_*=...` pairs and zero `-e CLAUDE_STATUS_LABEL_WAGGLE_*=...` pairs.
- `grep -n "CLAUDE_STATUS_LABEL_WAGGLE\|waggle_owned\|waggle_session_name\|waggle_model\|waggle_repo" src/claude_spawn/spawn.py` returns no matches as data values (docstring prose mentioning "waggle-owned workers" is acceptable — it ships in Epic 7).
- `list_spawned_workers_impl` and `answer_question_impl` both call `claude_status.workers(label="claude_spawn_owned=1")`.
- Both functions read the session name via `labels.get("claude_spawn_session_name", ...)` form.
