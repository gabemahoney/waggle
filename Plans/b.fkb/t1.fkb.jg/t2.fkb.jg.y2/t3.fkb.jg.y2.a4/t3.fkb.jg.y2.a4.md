## Context
The sting subcommand inspects `~/.claude.json` and `~/.claude/settings.json` for an `mcpServers` entry whose key matches a "looks like our project" regex. The regex was tuned for `waggle`-flavored keys; per SRD SR-7.2 it now matches `claude[-_]spawn`-flavored keys. The regex constant and its two helper functions are also renamed for symbol-name parity with the new project.

The pattern must:
- Match `claude-spawn`, `claude_spawn`, `my-claude-spawn`, `CLAUDE-SPAWN`, `claude_spawn_server`, `Claude-Spawn`.
- NOT match `claudespawn` (no separator), `claude-spawnfish` (no trailing separator), `myclaudespawn` (no leading separator), `claude-spawned` (no separator at end of stem).

## What Needs to Change
- In `src/claude_spawn/sting.py`:
  - Rename the module-level constant `_WAGGLE_PATTERN` to `_CLAUDE_SPAWN_PATTERN`. Update the compiled regex string from `(?i)(?:^|[-_])waggle(?:$|[-_])` to `(?i)(?:^|[-_])claude[-_]spawn(?:$|[-_])`. The pattern stays `re.compile(...)` with the `re.IGNORECASE` flag baked into the `(?i)` prefix.
  - Rename helper function `_key_matches_waggle(key: str) -> bool` to `_key_matches_claude_spawn(key: str) -> bool`. The body changes only to reference the renamed pattern constant.
  - Rename helper function `_has_waggle_in_mcp_servers(mcp_servers: object) -> bool` to `_has_claude_spawn_in_mcp_servers(mcp_servers: object) -> bool`. The body changes only to reference the renamed `_key_matches_*` helper.
  - Update every internal call site (specifically the two call sites inside `_detect_mcp`).
- DO NOT touch the `_detect_mcp` function's overall semantics, the file-path defaults (`~/.claude.json` and `~/.claude/settings.json`), or the file-not-found / JSON-decode-error / OS-error handling. The detection-target locations and error contract are unchanged per SR-7.1.
- DO NOT touch the `check_claude_status_health()` function or other module-level surface.

## Key Files
- `src/claude_spawn/sting.py` — pattern constant, two helper functions, two call sites in `_detect_mcp`.

## Acceptance Criteria
- `grep -n "_WAGGLE_PATTERN\|_key_matches_waggle\|_has_waggle_in_mcp_servers" src/claude_spawn/sting.py` returns no matches.
- `grep -n "_CLAUDE_SPAWN_PATTERN" src/claude_spawn/sting.py` returns at least three matches (the constant definition + the two helper bodies that call `.search` on it).
- `_CLAUDE_SPAWN_PATTERN.search("claude-spawn")`, `.search("claude_spawn")`, `.search("my-claude-spawn")`, `.search("CLAUDE-SPAWN")`, `.search("claude_spawn_server")`, `.search("Claude-Spawn")` all return non-None match objects.
- `_CLAUDE_SPAWN_PATTERN.search("claudespawn")`, `.search("claude-spawnfish")`, `.search("myclaudespawn")`, `.search("claude-spawned")` all return None.
- `_key_matches_claude_spawn` and `_has_claude_spawn_in_mcp_servers` exist with the same call-signature shape as the originals.
- `_detect_mcp()` semantics unchanged: returns True when an `mcpServers` key in either config file matches `_CLAUDE_SPAWN_PATTERN`; returns False on no-match, missing file, malformed JSON, or OS error.
