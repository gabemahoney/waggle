# Task 1: Rename sting MCP-detection pattern and helper functions

## Context
After Epics 1–5, the package name and binary are renamed; the env vars, label keys, tmux prefix, and MCP server name are all updated. The sting MCP-detection regex `_WAGGLE_PATTERN`, its helper functions `_key_matches_waggle` / `_has_waggle_in_mcp_servers`, and the corresponding `TestWagglePattern` test class still carry the legacy name. This Task renames them per SRD SR-7.

The pattern's regex body changes from `(?i)(?:^|[-_])waggle(?:$|[-_])` to `(?i)(?:^|[-_])claude[-_]spawn(?:$|[-_])` — the new form must match a `claude` literal followed by either `-` or `_` and then `spawn`, surrounded by `^`/`$`/`-`/`_`. Positive and negative test cases are explicitly enumerated in SR-7.2.

## What Needs to Change
- `src/claude_spawn/sting.py`:
  - Rename module-level constant `_WAGGLE_PATTERN` → `_CLAUDE_SPAWN_PATTERN`. Update the compiled regex to `(?i)(?:^|[-_])claude[-_]spawn(?:$|[-_])`.
  - Rename helper `_key_matches_waggle(key)` → `_key_matches_claude_spawn(key)`.
  - Rename helper `_has_waggle_in_mcp_servers(mcp_servers)` → `_has_claude_spawn_in_mcp_servers(mcp_servers)`.
  - Update every internal call site (`_detect_mcp` calls both helpers).
- `tests/test_sting.py`:
  - Flip the import names (`_WAGGLE_PATTERN` → `_CLAUDE_SPAWN_PATTERN`, etc.) in the existing `from claude_spawn.sting import (...)` block.
  - Rename `TestWagglePattern` → `TestClaudeSpawnPattern`. Rewrite its test methods with the SR-7.2 positive cases (`claude-spawn`, `claude_spawn`, `my-claude-spawn`, `CLAUDE-SPAWN`, `claude_spawn_server`, `Claude-Spawn`) and negative cases (`claudespawn`, `claude-spawnfish`, `myclaudespawn`, `claude-spawned`). Keep the same five test-method shape (exact / with suffix / with prefix / no-match-embedded / case-insensitive) so the test count is unchanged.
  - Update `TestDetectMcp` fixture JSON: positive `mcpServers` keys (currently `"waggle"`, `"waggle-server"`) become `claude-spawn`-rooted keys (e.g. `"claude-spawn"`, `"claude-spawn-server"`). Rename the method `test_no_waggle_key_returns_false` → `test_no_claude_spawn_key_returns_false`.

## Why
After this Epic the sting subcommand detects MCP-server registrations whose key matches the Claude Spawn pattern, not the Waggle pattern. The user's `mcpServers` config keys (set via `claude mcp add claude-spawn` or similar) are the operator-chosen key — the pattern just provides flexibility for variations like `my-claude-spawn`, `CLAUDE-SPAWN`, etc.

## Out of scope (deferred — DO NOT TOUCH)
- README + `docs/` prose — Epic 7 (`t1.fkb.xy`).
- The module docstring and `check_claude_status_health` docstring prose in `sting.py` that mention Waggle (already handled by Epic 2 Subtask D).

- `grep -RIn "_CLAUDE_SPAWN_PATTERN" src/claude_spawn/sting.py` returns at least three matches (definition + two helper bodies that call `.search` on it).
- The renamed `_CLAUDE_SPAWN_PATTERN` matches `claude-spawn`, `claude_spawn`, `my-claude-spawn`, `CLAUDE-SPAWN`, `claude_spawn_server`, `Claude-Spawn`; does NOT match `claudespawn`, `claude-spawnfish`, `myclaudespawn`, `claude-spawned`.
- `TestClaudeSpawnPattern` exists in `tests/test_sting.py` and exercises the new positive and negative cases.
- `TestDetectMcp` positive cases use `claude-spawn`-rooted `mcpServers` keys; negative case uses an unrelated key (e.g. `"other-tool"`).
- `poetry run pytest tests/test_sting.py` is fully green.
