## Context

The three new options `claude_home`, `claude_settings`, and `permissions` give orchestrators fine-grained control over the worker's Claude Code configuration. Without context on how Claude Code's settings layers compose, operators won't know which option to reach for or what "first-class `permissions` wins" means in practice. Depends on Task 3 landing.

## What Needs to Change

Add a new subsection to `README.md` titled **Claude Code settings stack** under the `spawn_worker` documentation. Cover:

- The four layers Claude Code merges, from lowest to highest precedence:
  1. Enterprise managed settings (e.g. `/etc/claude-code/managed-settings.json`).
  2. User settings — `<HOME>/.claude/settings.json`.
  3. Project shared settings — `<cwd>/.claude/settings.json`.
  4. Project local settings — `<cwd>/.claude/settings.local.json`.
- The CLI `--settings <path>` overlay wins above all four layers.
- How Claude Spawn's options map onto the stack:
  - `claude_home` rewrites `HOME` for the worker, redirecting layer 2 to a different config tree.
  - `claude_settings` supplies the path passed via `--settings` — the highest-precedence overlay.
  - `permissions` is realized inside that overlay: when no `claude_settings` is supplied, Claude Spawn synthesizes a minimal overlay containing just the `permissions` block; when both are supplied, the composite overlay layers the first-class `permissions.allow`/`deny`/`ask` on top of the caller's file (per SR-9.3, first-class `permissions` wins on those three keys; other top-level keys pass through unchanged).
- The SR-9.4 `--dangerously-skip-permissions` precedence note: passing `--dangerously-skip-permissions` via `claude_args` alongside a non-empty `permissions` map is not an error; Claude Code's CLI-level bypass wins at runtime.
