## Context

SR-6.5 introduces a new module `src/claude_spawn/templates.py` that owns all template I/O and schema validation. The module is consumed by Task 3 (`spawn_worker_impl` template resolution) and by Epic 4 (`list_templates` MCP tool). It must follow the project's "import-time has no filesystem reads" convention (see `src/claude_spawn/claude_status.py` for the established pattern) so test fixtures can install patches before any I/O occurs. At this subtask's boundary the loader is dead code — Task 3 wires it into `spawn_worker_impl`.

## What Needs to Change

Create `src/claude_spawn/templates.py` with the following shape:

- **Filesystem seams (verbatim per SR-6.5):** "All filesystem reads go through a single, patchable seam." Two patch targets:
  - `_templates_dir()` returns the templates directory (default `~/.claude-spawn/templates/`, expanded via `os.path.expanduser`).
  - `_read_template_file(path)` reads and returns the raw text of one template file. This is the single seam through which all template content enters the module.
- **Import inertia (verbatim per SR-6.5):** "Import of `src/claude_spawn/templates.py` performs no filesystem reads." Module-level code must not call `_templates_dir()` or `_read_template_file()`; both are called only from the public functions.
- **Public API (verbatim per SR-6.5):** "The loader exposes at least these functions: one to resolve and read a single template by name (used by `spawn_worker_impl`) and one to enumerate all templates (used by `list_templates_impl`)."
  - Single-template lookup: takes a template name string; returns the parsed option-map dict on success; returns a structured error result for `ErrTemplateNotFound` (file absent — description names both the template name AND the templates directory) and `ErrTemplateMalformed` (validation failure — description names the file path and the specific problem).
  - Enumerate-all: yields one entry per `.toml` file in the templates directory. Each entry carries `(name, path, parsed_options_or_error)`. Malformed files are surfaced via the error half — they do NOT abort the iteration. Missing directory yields zero entries with no error.
- **Schema validation per SR-6.4:** A template is malformed (→ `ErrTemplateMalformed`) if any of:
  - The file fails to parse as TOML.
  - The top-level table contains a key outside the SR-1.1 schema (the eleven option names minus `template`).
  - The top-level table contains a `template` key (SR-2.4 — templates cannot reference templates).
  - `thinking` is present and is not one of `low`/`medium`/`high`/`xhigh`.
  - A scalar option's value is not a TOML string.
  - A map option's value is not a TOML table.
  - A list option's value is not a TOML array of strings.
- **Use `tomllib` from the standard library** for parsing. Do NOT use `tomli-w` here (write path is Epic 5).

No changes to `src/claude_spawn/spawn.py` or any other source file in this subtask.

## Key Files

- `src/claude_spawn/claude_status.py` — read-only; reference for the "import-time no I/O" convention.

- `import claude_spawn.templates` completes without any filesystem reads (verifiable by mocking both seams and confirming neither was called during import).
- The single-template lookup returns the parsed option map on success.
- The single-template lookup surfaces `ErrTemplateNotFound` (file absent) with description naming both the template name and the templates directory.
- The single-template lookup surfaces `ErrTemplateMalformed` (each SR-6.4 condition) with description naming the file path and the specific problem.
- The enumerate-all function yields one entry per `.toml` file, with malformed files in the error half rather than aborting iteration.
- Missing templates directory: enumerate-all yields zero entries, no error.
- `src/claude_spawn/spawn.py` is NOT modified by this subtask.
- `tomli-w` is NOT imported here.
