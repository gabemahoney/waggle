## Context

This subtask layers the `--interactive` mode on top of the flag-driven subcommand from the sibling subtask. Per SR-7.5, interactive mode walks through every option (except `template`) with one prompt at a time, showing the current resolved value if a template by that name already exists, and accepts either a value or the literal token `skip`. Interactive mode is mutually exclusive with flag-driven mode. Both modes delegate to a single `write_template_impl` so they produce identical files for identical inputs.

## What Needs to Change

In `src/claude_spawn/cli.py`:
- Add `--interactive` to the mutually-exclusive argparse group scaffolded by the sibling subtask. Passing `--interactive` together with any option flag (e.g. `--cwd`) is rejected at the argparse level via the mutually-exclusive group; the parser's standard JSON error envelope is emitted on stdout with exit 2.
- When `args.interactive` is true, invoke a new helper (e.g. `_interactive_collect_options(name)`) instead of assembling `options` from flags. The helper:
  - Iterates through every option from SR-1.1 except `template` in a fixed order.
  - For each option: loads the existing template (if any) so the current resolved value can be displayed. Prints the prompt (option name + short description + current value if any) to STDERR — NOT stdout, so the success-line JSON output remains uncontaminated. Reads one line from stdin via `builtins.input`.
  - If the operator types the literal token `skip`, omits that option from the collected dict.
  - For list-valued options (`claude_args`) and map-valued options (`extra_env`, `claude_status_labels`, `permissions.allow`/`deny`/`ask`), the helper accepts entries in a sub-loop and terminates when the operator types `skip` (or an equivalent termination token), accumulating entries.
- Wrap the helper call (and the subsequent `write_template_impl` dispatch) in a `try/except (KeyboardInterrupt, EOFError)` block. On cancellation: print `json.dumps({"status":"error","message":"write-template cancelled"})` to stdout and exit with a non-zero code (2). NO call to `write_template_impl`; NO file written.
- After `_interactive_collect_options` returns successfully, the dispatch path is identical to flag-driven mode: call `write_template_impl(name, options, force)` and handle success/error the same way.

## Key Files

- `src/claude_spawn/cli.py` — `--interactive` flag wiring, interactive helper, cancellation handler.
- `src/claude_spawn/templates.py` — read-only; `write_template_impl` and the loader's lookup function used for displaying current values.

## Acceptance Criteria

- `claude-spawn write-template foo --interactive` with `builtins.input` patched to return values writes a file and prints a success JSON line on stdout with exit 0.
- The file produced by interactive mode for a given input set is byte-identical to the file produced by flag-driven mode for the same input set.
- A `skip` reply omits that option from the written file.
- On `EOFError` from `builtins.input` (or `KeyboardInterrupt`): stdout contains exactly `{"status":"error","message":"write-template cancelled"}`; exit code is non-zero (typically 2); no file is written.
- `claude-spawn write-template foo --interactive --cwd /tmp` produces an argparse-level JSON error on stdout with exit 2 and writes no file (mutually exclusive group enforcement).
- When a template named `foo` already exists, each interactive prompt for an option that is set in that template displays the current resolved value before reading input.
- Prompts go to STDERR; the success line is the only thing on STDOUT.
