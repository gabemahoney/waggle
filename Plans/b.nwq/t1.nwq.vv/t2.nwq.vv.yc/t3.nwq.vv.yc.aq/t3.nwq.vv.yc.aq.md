## Context

SR-7.3 introduces a new programmatic authoring surface: a function that writes a template file from a dict of options, with name safety, schema validation, atomic write, and a `force` flag. Both the MCP tool (sibling subtask) and the future CLI subcommand (Epic 6) MUST delegate to this single function so the two surfaces are guaranteed to produce identical files for identical inputs. The function lives in `src/claude_spawn/templates.py` alongside the Epic 3 loader.

## What Needs to Change

In `src/claude_spawn/templates.py`, add `write_template_impl(name, options, force)`:

- **Name safety (SR-7.4):** reject the name if it contains a path separator (`/` on POSIX), starts with `.`, equals `..`, contains `..` as a path-component substring, or is empty. Return `{"ok": False, "err_name": "ErrTemplateNameUnsafe", "err_description": "..."}` on any of these. No file is written.
- **Options validation (SR-6.4 rules, reused at write time):** unknown keys (outside the SR-1.1 schema minus `template`), a `template` key (SR-2.4), an invalid `thinking` level (not in `low`/`medium`/`high`/`xhigh`), and type mismatches (scalar not a string, map not a dict, list not a list of strings) all return `{"ok": False, "err_name": "ErrTemplateOptionsInvalid", "err_description": "<enumeration of rejected keys/values>"}`. No file is written. Validation reuses the same logic the loader applies — extract to a shared helper if it does not already exist.
- **Directory auto-creation:** if the templates directory (from `_templates_dir()`) does not exist, create it (including any missing parents).
- **Defensive path-construction check (verbatim per SR-7.4):** "Defensive check after path construction (e.g. resolving the parent and confirming it equals the templates directory) is required." After constructing the on-disk path as `<templates_dir>/<name>.toml`, resolve its parent and assert it equals the resolved templates directory. Any mismatch → `ErrTemplateNameUnsafe`.
- **Collision check:** if the canonical path already exists and `force` is false, return `{"ok": False, "err_name": "ErrTemplateExists", "err_description": "<existing path>"}` without modifying the file.
- **Atomic write (verbatim per Epic body):** "Atomic write via temp + rename within the same directory." Serialize `options` with `tomli-w`, write to a sibling temp file in the same directory, then `os.rename` to the canonical path. This ensures a crash between the temp write and the rename leaves the canonical file unchanged (if it existed) or absent (if it did not) — never partially written.
- **Success return:** `{"ok": True, "path": "<absolute path>", "options": <normalized options>}`.

The function MUST raise no unhandled exceptions for expected error conditions; unexpected exceptions bubble up and are caught by the MCP wrapper.

## Key Files

- `src/claude_spawn/templates.py` — new `write_template_impl` function.
- `pyproject.toml` — read-only; `tomli-w` already a runtime dep from Epic 3.

## Acceptance Criteria

- `write_template_impl("orch", {"cwd": "/tmp"}, force=False)` against an empty `fake_templates_dir` returns `{"ok": True, "path": "...orch.toml", "options": {"cwd": "/tmp"}}`; the file exists; loading it through the Epic 3 loader returns the same options.
- Second call with same name and `force=False` returns `ErrTemplateExists`; file content unchanged.
- Second call with `force=True` succeeds, overwrites atomically (file content reflects new options).
- Each of the five unsafe-name shapes returns `ErrTemplateNameUnsafe`.
- Each of the four invalid-options conditions returns `ErrTemplateOptionsInvalid` with description naming the offending key/value.
- Directory auto-creates when absent.
- `write_template_impl` is callable by both the MCP wrapper (sibling subtask) and the future CLI (Epic 6) — i.e. it is a sync function, not a class method or async coroutine.
