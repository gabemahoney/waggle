## Context

Epic 3 created `docs/architecture/spawn_options_and_templates.md`. Epic 4 extended it with `list_templates`. This subtask adds a `write_template` write-path subsection covering the validation order, the defensive path-construction check, and the atomic temp+rename write pattern.

## What Needs to Change

In `docs/architecture/spawn_options_and_templates.md`, append a new section (e.g. "Template authoring: write_template") covering:

- The full write path in order: SR-7.4 name safety → SR-6.4 options validation → directory auto-creation → defensive parent-equality path check → atomic write (`tomli-w` serializes to a sibling temp file in the same directory, then `os.rename` to the canonical path).
- Atomic-rename property: a crash between the temp write and the rename leaves the canonical path untouched if the file already existed, or absent if it did not. The canonical file is never partially written.
- Loader-shared-path observation: `write_template_impl` writes to the same on-disk path that the loader (Epic 3) reads from. A freshly written template is immediately visible to the next `spawn_worker(template=<name>)` call (no reload required; SR-6.6's no-caching property amplifies this).
- Single-impl note: `write_template_impl` is the shared implementation both the MCP tool (Epic 5) and the future CLI subcommand (Epic 6) delegate to. Forward reference to Epic 6.

Out of scope: CLI invocation examples (Epic 6).
