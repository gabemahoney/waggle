# Epic 2 — Spawn-readiness blocking with timeout cleanup

Make `spawn_worker_impl` block until the worker has registered itself with Claude Status, so the returned `instance_id`/`tmux_session_name` can be used immediately for `send_input`/`get_output`/etc.

## Scope

- **Blocking semantics (SR-5.1):** `spawn_worker_impl` returns only after a worker with the spawned `instance_id` appears in Claude Status's `instances` table, or after the readiness timeout elapses. The poll uses `claude_status.workers(label="claude_spawn_owned=1")` via the existing `_run` subprocess seam — no new seam introduced — and matches by exact `instance_id` equality (SR-10.1). Poll interval short enough that happy-path return latency is bounded by actual registration time + at most one poll round-trip.
- **Timeout default (SR-5.2):** module-level constant in `src/claude_spawn/spawn.py` set to 15 seconds. Not exposed to callers, not template-overridable in v1.
- **Timeout cleanup (SR-5.3):** on timeout, issue `tmux kill-session -t <tmux_session_name>` before returning, then return `ErrSpawnReadinessTimeout` with a description naming the `tmux_session_name` and the timeout value. Kill is best-effort; if it fails the error name is unchanged.
- **Worker-exited-early detection (SR-5.4):** during the readiness wait, interleave `tmux has-session -t <tmux_session_name>` probes with the Claude Status polls. On detected early exit, capture pane content via `tmux capture-pane` (best-effort, while still capturable) and return `ErrSpawnWorkerExitedEarly` with the captured output in the description.
- **Tests:** new `tests/test_engine_spawn_worker_readiness.py` covering:
  - Timely registration: fake Claude Status returns the worker on poll N; `spawn_worker_impl` returns the success dict with no `kill-session` call.
  - Registration timeout: fake Claude Status never returns the worker; assert `ErrSpawnReadinessTimeout`, assert `kill-session -t <tmux_session_name>` was issued. The test must run fast — use a patched timeout constant or a monkeypatched `time.monotonic()`/poll-loop accelerator rather than waiting 15 real seconds.
  - Worker-exited-early: `has-session` probe returns non-zero; assert `ErrSpawnWorkerExitedEarly`, `capture-pane` was attempted before the session died, error description includes captured content.
  - Uses `tests.helpers.fake_claude_status` (multi-triple sequence for the poll loop, no signature change required per SR-11.3).
- **Documentation:** README updated to note that `spawn_worker` blocks until registration (default 15s timeout); on success callers may immediately use the returned `tmux_session_name`. `docs/architecture/spawn_worker.md` updated with the readiness wait in the data-flow description.

## Out of scope for this Epic

- Per-call or template-level override of the readiness timeout (SR-13).
- Any changes to template loading (Epic 3).

## Acceptance Criteria

1. **Happy path.** Configure `fake_claude_status` so the spawned worker's `instance_id` appears on the second poll. Call `spawn_worker_impl(cwd="/tmp")` (via stubbed tmux). Assert: return value is the `{instance_id, tmux_session_name}` dict; no `tmux kill-session` call was issued; the readiness loop ran ≥1 and ≤2 polls.
2. **Timeout path.** Configure `fake_claude_status` so the worker never appears. With the readiness timeout patched down to a small value (e.g. 0.2s) so the test runs fast, call `spawn_worker_impl(cwd="/tmp")`. Assert: returns `err_name=ErrSpawnReadinessTimeout`; description names both the `tmux_session_name` and the timeout; the captured `_tmux` calls include `kill-session -t <tmux_session_name>`.
3. **Best-effort kill.** Same as #2 but with `_tmux` configured to raise on `kill-session`. Assert: the returned `err_name` is still `ErrSpawnReadinessTimeout` (kill failure does not change the error surface).
4. **Worker-exited-early path.** Configure `_tmux` so the second `has-session` probe returns non-zero (session gone), and `capture-pane` returns a known string `"FATAL: something"`. Call `spawn_worker_impl(cwd="/tmp")`. Assert: returns `err_name=ErrSpawnWorkerExitedEarly`; description contains `"FATAL: something"`.
5. **Test suite green.** `pytest tests/test_engine_spawn_worker_readiness.py` plus the full suite pass. Existing tests from Epic 1 continue to pass once they have been updated (where needed) to satisfy the new readiness check — Epic 1's tests should already arrange `fake_claude_status` to return the worker on the first poll for the happy-path tmux launch assertions, since the readiness wait is now part of the spawn contract.
6. **Docs.** README spawn section describes the readiness-blocking behavior and the 15-second default; `docs/architecture/spawn_worker.md` data-flow diagram or prose shows the poll/check/cleanup loop.
