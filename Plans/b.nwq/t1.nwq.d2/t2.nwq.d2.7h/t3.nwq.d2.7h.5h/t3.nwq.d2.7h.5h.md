## Context

Epic 2's engineer subtask adds a polling loop after the tmux `send-keys` call in `spawn_worker_impl`. Every happy-path test in Epic 1's test files (`tests/test_engine_spawn_worker.py` and `tests/test_engine_spawn_worker_options.py`) that previously asserted on a successful return will now hang or time out, because the new readiness loop polls Claude Status forever until the spawned `instance_id` appears. This subtask updates those tests to wrap each happy-path `spawn_worker_impl` call in a `fake_claude_status` context whose first triple returns a `fake_workers_response` containing the spawned worker — so the readiness loop finds the worker on the first poll and the test runs in milliseconds.

This subtask does NOT include any signature, return-key, or label-rename work — all of that landed in Epic 1.

## What Needs to Change

In `tests/test_engine_spawn_worker.py` AND `tests/test_engine_spawn_worker_options.py`:
- For every happy-path test that calls `spawn_worker_impl` and asserts on a success result, wrap the call in a `fake_claude_status([...])` context whose triples include at least one `fake_workers_response([fake_worker_record(instance_id=<spawned id>, status="working", cwd="/tmp")])` so the readiness poll matches on the first iteration.
- Because `instance_id` is auto-generated inside `spawn_worker_impl` when the caller does not pass one, the pattern is: either (a) pass an explicit `instance_id=...` argument so the test can pre-build the matching worker record, or (b) match against any worker the test arranges, relying on the impl's exact-equality scan. Choose whichever is clearer per test.
- Update any test assertions that count `_tmux` calls — the readiness loop adds at least one `has-session` probe on the happy path, so legacy assertions of "exactly two `_tmux` calls" become "at least two" or an explicit list with the additional probe(s).
- Do NOT wrap error-path tests (those that assert on `err_name` for SR-9.1 errors) — those returns short-circuit before the readiness loop runs, and adding a `fake_claude_status` wrapper would be misleading.

## Key Files

- `tests/test_engine_spawn_worker.py` — wrap happy-path tests with `fake_claude_status`.
- `tests/test_engine_spawn_worker_options.py` — same.

## Acceptance Criteria

- `pytest tests/test_engine_spawn_worker.py tests/test_engine_spawn_worker_options.py -v` passes with 0 failures, 0 errors after Epic 2's engineer subtask has landed.
- No happy-path test hangs or exceeds a reasonable per-test timeout (e.g. `pytest --timeout=5` succeeds).
- Error-path tests remain un-wrapped (no `fake_claude_status` context).
- All `_tmux` call-count assertions reflect the new post-readiness-loop counts.
- No test uses old label keys, the old `session_name` return key, or any pre-Epic-1 signature shape (those were already updated in Epic 1).
