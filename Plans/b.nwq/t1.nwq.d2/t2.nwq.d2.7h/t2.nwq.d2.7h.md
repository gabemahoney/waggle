# Task 1 — Implement spawn-readiness blocking with timeout cleanup and worker-exited-early detection

## Context

After Epic 1, `spawn_worker_impl` returns as soon as the tmux `send-keys` succeeds — but the worker has not yet registered with Claude Status, so callers cannot immediately use `send_input`/`get_output`/etc. against the returned `tmux_session_name`. SR-5 makes the return blocking: `spawn_worker` polls Claude Status's `instances` table until the spawned `instance_id` appears, or until the 15-second readiness timeout (SR-5.2) expires. SR-5.3 mandates `tmux kill-session` cleanup on timeout. SR-5.4 mandates detection of worker-exited-early via `tmux has-session` probes interleaved with the poll, with `tmux capture-pane` diagnostics on detected exit.

## What Needs to Change

- Add a module-level `_SPAWN_READINESS_TIMEOUT = 15` constant to `src/claude_spawn/spawn.py` (SR-5.2). Not exposed to callers or templates in v1.
- After the existing tmux `send-keys` block in `spawn_worker_impl`, run a polling loop that polls `claude_status.workers(label="claude_spawn_owned=1")` via the existing `_run` seam, matches against the spawned `instance_id` by exact equality (SR-10.1), interleaves a `tmux has-session` probe to detect early exit, and returns the success dict `{instance_id, tmux_session_name}` as soon as the worker registers.
- On timeout: issue best-effort `tmux kill-session -t <tmux_session_name>` (kill failure does NOT change the returned `err_name`), then return `err_name=ErrSpawnReadinessTimeout` with description naming both `tmux_session_name` and the timeout value.
- On worker-exited-early: capture pane content via `tmux capture-pane` (best-effort, while still capturable), then return `err_name=ErrSpawnWorkerExitedEarly` with captured output in the description.
- Tests:
  - New `tests/test_engine_spawn_worker_readiness.py` covering all four paths (happy, timeout, best-effort-kill, worker-exited-early).
  - Adapt Epic 1's existing happy-path tests (`tests/test_engine_spawn_worker.py`, `tests/test_engine_spawn_worker_options.py`) to wrap calls in `fake_claude_status` so they don't block on the new readiness loop.
  - Full pytest run at the end.

## Why

Lets orchestrators rely on the post-spawn `tmux_session_name` immediately — closes the gap between "tmux session created" and "Claude Status sees the worker". The cleanup behavior on timeout/early-exit ensures the operator's host doesn't accumulate orphaned tmux sessions.

## Success Criteria

- `spawn_worker_impl` returns success only after the spawned `instance_id` appears in Claude Status.
- On timeout, `tmux kill-session` is issued and the return is `ErrSpawnReadinessTimeout` with the timeout value and `tmux_session_name` in the description.
- On detected early exit, `tmux capture-pane` is attempted and the return is `ErrSpawnWorkerExitedEarly` with the captured output in the description.
- `_SPAWN_READINESS_TIMEOUT = 15` exists at module level in `spawn.py` and is not exposed through any caller-facing parameter or template field.
- The poll uses `claude_status.workers(label="claude_spawn_owned=1")` via the existing `_run` seam; no new subprocess seam is added.
- Full `pytest` is green, including the new readiness test file.
