## Context

`docs/architecture/spawn_worker.md` documents `spawn_worker_impl`'s data flow with a parameters table, a flow section, an errors table, and a Mermaid sequence diagram. Epic 1 already updated the parameters table and the launch-composition portion of the flow. Epic 2 adds a poll/probe/cleanup loop between the existing `send-keys` call and the final return; the doc must show that loop and surface the two new error names.

## What Needs to Change

In `docs/architecture/spawn_worker.md`:

- **Flow section:** insert a "Readiness wait" step (or expand the final step) describing the loop:
  1. Poll `claude_status workers --label claude_spawn_owned=1` via the existing `_run` seam.
  2. On match by exact `instance_id` equality: return success `{instance_id, tmux_session_name}`.
  3. On `tmux has-session` probe returning non-zero (interleaved with the poll): attempt `tmux capture-pane` (best-effort) and return `ErrSpawnWorkerExitedEarly` with captured output in the description.
  4. On timeout (the module-level `_SPAWN_READINESS_TIMEOUT` constant, **15 seconds**): issue best-effort `tmux kill-session`, then return `ErrSpawnReadinessTimeout` with the `tmux_session_name` and the timeout value in the description.
- **Mermaid sequence diagram:** extend the diagram to show the readiness-wait loop with three exit edges (success, timeout → kill-session → error, early-exit → capture-pane → error). The diagram should make clear that the readiness wait happens AFTER `send-keys` succeeds, not before.
- **Errors table:** add two new rows for `ErrSpawnReadinessTimeout` and `ErrSpawnWorkerExitedEarly` alongside the existing entries from Epic 1.

Out of scope: do NOT create `docs/architecture/spawn_options_and_templates.md` (Epic 3 owns that). Do NOT describe templates, `list_templates`, `write_template`, or the CLI.
