# SRD — Rename Waggle → Claude Spawn

Companion to the PRD at `t1.hys.e8`. Requirements describe what must be true about the software after the refactor. Implementation is out of scope.

## SR-1: Package and Distribution Identity

### SR-1.1: Distribution metadata

- `pyproject.toml` `[tool.poetry] name` is the kebab-case literal `claude-spawn`.
- `pyproject.toml` `[tool.poetry] description` no longer contains the substring "Waggle" (case-insensitive).
- `pyproject.toml` `[tool.poetry] packages` declares the single include `{include = "claude_spawn", from = "src"}`.
- `pyproject.toml` `[tool.poetry.scripts]` defines exactly one entry: `claude-spawn = "claude_spawn.cli:main"`. The legacy `waggle` script key is absent.

### SR-1.2: Python package layout

- The source package directory is `src/claude_spawn/` (snake_case).
- The directory contains the same set of modules that `src/waggle/` contained before the refactor: `__init__.py`, `cli.py`, `mcp_stdio.py`, `spawn.py`, `claude_status.py`, `sting.py`, `installer.py`.
- No `src/waggle/` directory exists in the working tree.
- `src/claude_spawn/__init__.py`'s module docstring and `__version__` attribute exist; the docstring uses "Claude Spawn" in place of any prior "Waggle" mention.

### SR-1.3: Import surface

- `import claude_spawn` and `from claude_spawn import <submodule>` succeed in a clean environment with the package installed.
- `import waggle` raises `ModuleNotFoundError`.
- Every internal cross-module import in `src/claude_spawn/` references `claude_spawn.X`, never `waggle.X`.

### SR-1.4: CLI binary

- After `pip install .` (or `poetry install`), the executable `claude-spawn` is on `PATH`.
- The executable `waggle` is not on `PATH` (no shim, no alias).

## SR-2: MCP Server Identity

### SR-2.1: Advertised server name

- The stdio MCP server constructed by `claude_spawn.mcp_stdio` advertises `serverInfo.name = "claude-spawn-stdio"` in its MCP `initialize` response.
- The literal `waggle-stdio` does not appear in `claude_spawn.mcp_stdio`.

### SR-2.2: Recommended operator-installed key

- The README's `claude mcp add` example uses `claude-spawn` as the operator-chosen `mcpServers.<KEY>` value.
- No documentation example or default uses `waggle` as the `mcpServers.<KEY>` value.

## SR-3: CLI Surface

### SR-3.1: Program identity

- `claude-spawn --help` exits 0 and prints program description text that includes "Claude Spawn" and does not include "Waggle" (case-insensitive).
- The argparse `prog` for the top-level parser is `claude-spawn`.
- The custom argparse subclass (currently `WaggleArgumentParser`) is renamed to `ClaudeSpawnArgumentParser`; its behavior — emitting `{"status": "error", "message": ...}` to stdout and exiting 2 on usage errors — is unchanged.

### SR-3.2: Subcommand surface

- Three subcommands are exposed, with the same dispatch semantics as before the refactor: `sting`, `mcp`, `install`.
- The set of subcommands is unchanged; no add, no remove, no rename.
- Per-subcommand `help`, `description`, and option-`help` strings reference "Claude Spawn" / `claude-spawn`, never "Waggle" / `waggle`.

### SR-3.3: `--auq-order` flag

- `claude-spawn install --auq-order <value>` forwards the value to `claude-status install-hooks --auq-order` unchanged.
- The flag's `help` text uses `claude-spawn` (or omits the name) in any example values; it must not reference `waggle` (e.g. `before:waggle`).

## SR-4: Environment-Variable Injection and Label Schema

### SR-4.1: Env vars set on tmux new-session

- `claude_spawn.spawn.spawn_worker_impl` sets exactly seven environment variables on the tmux `new-session` invocation (via `-e KEY=val` arguments):
  - `CLAUDE_STATUS_INSTANCE_ID` — the generated UUID for the worker
  - `CLAUDE_STATUS_RELAY_MODE` — fixed value `on`
  - `CLAUDE_STATUS_AUQ_MODE` — fixed value `record`
  - `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_OWNED` — fixed value `1`
  - `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_SESSION_NAME` — the resolved session name
  - `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_MODEL` — the model argument, lowercased
  - `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_REPO` — the repo argument, verbatim
- The four `CLAUDE_STATUS_LABEL_*` env vars use the prefix `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_`. No env var name in the spawn surface contains the substring `WAGGLE` (case-insensitive).

### SR-4.2: `_REQUIRED_ENV_VARS` contract

- The module-level `_REQUIRED_ENV_VARS` tuple in `claude_spawn.spawn` enumerates exactly the seven env var names listed in SR-4.1.
- Order within the tuple is stable across the refactor for test assertions that iterate the tuple.

### SR-4.3: Resulting Claude Status label keys

- A worker spawned by `claude_spawn.spawn.spawn_worker_impl` produces a Claude Status `instances.labels` map whose keys are exactly: `claude_spawn_owned`, `claude_spawn_session_name`, `claude_spawn_model`, `claude_spawn_repo`.
- No live code path writes `waggle_*` label keys.

### SR-4.4: Non-label env vars unchanged

- The three Claude-Status-owned env var names (`CLAUDE_STATUS_INSTANCE_ID`, `CLAUDE_STATUS_RELAY_MODE`, `CLAUDE_STATUS_AUQ_MODE`) and their values (`on`, `record` for the latter two) are unchanged.

## SR-5: Claude Status Worker Filter

### SR-5.1: `list_spawned_workers` filter

- `claude_spawn.spawn.list_spawned_workers_impl` invokes `claude-status workers --label claude_spawn_owned=1` (via `claude_spawn.claude_status.workers(label=...)`).
- The projected output continues to return `{"workers": [{"instance_id": str, "session_name": str}, ...]}` with `session_name` sourced from `labels.claude_spawn_session_name`.
- Skipped rows are still logged to stderr and excluded from the returned list; the stderr log line uses the new project name in any prose, but log content is otherwise unchanged in shape.

### SR-5.2: `answer_question` filter and label lookup

- `claude_spawn.spawn.answer_question_impl` queries Claude Status with `claude-status workers --label claude_spawn_owned=1`.
- After identifying the matched worker, the implementation reads `labels.claude_spawn_session_name` (not `labels.waggle_session_name`) to determine the tmux session to target.

## SR-6: Tmux Session Naming

### SR-6.1: Default session-name generation

- When `session_name=None` is passed to `claude_spawn.spawn.spawn_worker_impl`, the generated default is `spawn-<first8>`, where `<first8>` is the first 8 characters of the worker's UUID (UUID4 hex, hyphens included as in `uuid.uuid4()` string form).
- No default-generated session name has the prefix `waggle-`.

### SR-6.2: Caller-provided session name pass-through

- When the caller passes a non-`None` `session_name`, that value is used verbatim as the tmux session name and as the value of the `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_SESSION_NAME` env var. The pass-through is unchanged from prior behavior.

## SR-7: Sting MCP-Detection

### SR-7.1: Detection-target locations

- `claude_spawn.sting._detect_mcp()` scans the top-level `mcpServers` dict in two paths only: `~/.claude.json` and `~/.claude/settings.json`. Both paths are unchanged from prior behavior.
- File-not-found, JSON-decode-error, and OS-error conditions for either file are treated as "not detected" (i.e. the scan proceeds to the next file or returns `False`), unchanged from prior behavior.

### SR-7.2: Detection pattern

- The module-level pattern constant is named `_CLAUDE_SPAWN_PATTERN` and is a compiled regex equivalent to `(?i)(?:^|[-_])claude[-_]spawn(?:$|[-_])`.
- The pattern matches operator-chosen keys including (non-exhaustive): `claude-spawn`, `claude_spawn`, `my-claude-spawn`, `CLAUDE-SPAWN`, `claude_spawn_server`, `Claude-Spawn`.
- The pattern does not match: `claudespawn` (no separator), `claude-spawnfish` (no trailing separator), `myclaudespawn` (no leading separator), `claude-spawned` (no separator at end of stem).
- The pattern does not match any key containing only the substring `waggle`.

### SR-7.3: Helper symbol names

- The helper functions `_key_matches_claude_spawn(key)` and `_has_claude_spawn_in_mcp_servers(mcp_servers)` exist in `claude_spawn.sting` with the same call-signature and semantics as their pre-refactor counterparts.
- No `_WAGGLE_PATTERN`, `_key_matches_waggle`, or `_has_waggle_in_mcp_servers` symbol exists in `claude_spawn.sting`.

### SR-7.4: `claude-spawn sting` exit semantics

- `claude-spawn sting` exits 0 when Claude Status is reachable and its `capabilities` response advertises `contract_version` major `1`; otherwise it exits 1 with an actionable error message on stderr. The exit semantics are unchanged.
- Any prose in success/error messages references "Claude Spawn" / `claude-spawn`, never "Waggle" / `waggle`.

## SR-10: Preserved Behavior

### SR-10.1: MCP tool surface

- The six MCP tools exposed by `claude_spawn.mcp_stdio` retain their names verbatim: `spawn_worker`, `list_spawned_workers`, `terminate_worker`, `send_input`, `get_output`, `answer_question`.
- Each tool's parameter names, parameter types, return-value shape, and SR-7.1 operation-failed error envelope keys (`ok`, `operation`, `err_name`, `err_description`) are unchanged.
- The `scrollback` range `[1, 1000]` and default `50` for `get_output` are unchanged.
- The "no implicit Enter" semantics of `send_input` are unchanged.
- The `answer_question` single-question constraint (refuse with `ErrMultiQuestionUnsupported` when the pending AUQ has more than one question) is unchanged.
- The `answer_question` visibility check (whitespace-normalized substring match between question text and captured pane) is unchanged.

### SR-10.2: Subprocess seams

- `claude_spawn.spawn._tmux` is the only tmux subprocess seam used by spawn-module implementations. Its argv shape, return triple `(stdout, stderr, returncode)`, and 10-second timeout are unchanged.
- `claude_spawn.claude_status._run` is the only `claude-status` subprocess seam used by the claude_status client. Its argv shape and return contract are unchanged.

### SR-10.3: Installer behavior

- `claude_spawn.installer.handle_install`:
  - Verifies `claude-status` is on `PATH` before doing anything else; exits 1 with an actionable error if absent.
  - Builds an env dict that includes `CLAUDE_STATUS_RELAY_MODE=on` and `CLAUDE_STATUS_AUQ_MODE=record`.
  - Invokes `claude-status install-hooks` via the `_run_install_hooks` seam and forwards an optional `--auq-order` value.
  - On non-zero exit code, exits 1 with the stderr/stdout error string.
  - On success, prints any stdout (stripped) and calls `_remove_legacy_hook_template()` before exiting 0.
- `_remove_legacy_hook_template()` continues to delete `hooks/settings.json.template` if it exists, treating missing-file as a no-op. The function is preserved.

### SR-10.4: Claude Status consumer-CLI client

- `claude_spawn.claude_status` continues to provide `workers(label=...)`, `worker(instance_id=...)`, and `capabilities()` callables whose return shape and error-envelope structure are unchanged. Only the patch-target path string changes (see SR-12).

### SR-10.5: Sting CLI reference helper

- `claude_spawn.sting.check_claude_status_health()` returns a `(healthy: bool, message: str)` tuple. Its three "green" conditions (claude-status on PATH, capabilities succeeds, contract_version major 1) and the corresponding red messages are unchanged in semantics. Only prose tokens referencing the project name change.

## SR-13: Non-Functional Requirements

### SR-13.1: Performance

- `claude-spawn --help` and `claude-spawn sting` complete within the same time budget as their pre-refactor equivalents (sub-second on a modern host with Claude Status installed).
- No new subprocess, file-system scan, or network call is introduced on any code path; only string/identifier substitutions and dead-code deletions are performed.

### SR-13.2: Deployment

- Explicitly omitted. There is no deployment pipeline as part of this refactor. The previous `install.sh` + systemd-service deployment surface is deleted (SR-8.1); no replacement is built. Operator installs via `pip install . && claude-spawn install`.

### SR-13.3: HTTP / network surface

- Explicitly omitted. Claude Spawn exposes no HTTP server, no socket listener, no network ingress. The stdio MCP transport over `stdin`/`stdout` is the only IPC surface. Outbound calls are limited to `tmux` and `claude-status` subprocess invocations, both unchanged by this refactor.

### SR-13.4: Authentication / authorization

- Explicitly omitted. Claude Spawn has no auth surface of its own. Permission decisions are delegated to Claude Status via its `decide` subcommand (out of process). This is unchanged by the refactor.

### SR-13.5: Security

- Explicitly omitted as a domain of new work. The refactor introduces no new attack surface: no new subprocess invocations, no new file reads or writes, no new env-var reads, no new network calls. The set of env vars the spawn module emits is renamed but unchanged in count and value-shape; the `claude-status` subprocess argv shape is unchanged.

### SR-13.6: Data models / schemas

- The data-model changes are entirely the label-key rename (SR-4.3): `instances.labels` keys change from `waggle_*` to `claude_spawn_*`. Claude Status's own schema is unchanged. No persistent storage owned by Claude Spawn exists.

### SR-13.7: Testing strategy

- Test coverage is preserved at parity (212 tests; SR-11.1). The strategy is unchanged from pre-refactor: subprocess seams are patched at the module path via `unittest.mock.patch`, canonical sample payloads come from `tests/sample_payloads.py`, and helpers come from `tests/helpers.py`. See SR-11 and SR-12 for the concrete contract.

## Out of Scope

- Functional changes to surviving MCP tools, CLI subcommands, or runtime semantics (see PRD non-goals).
- Modifying Claude Status code or schemas; only the labels Claude Spawn writes change.
- Rebuilding a CI pipeline or installer for the new stdio-MCP surface; the docker/CI flow and `install.sh` are deleted, not replaced.
- Coordinating an external operator migration. The only operator is the local dev host.
