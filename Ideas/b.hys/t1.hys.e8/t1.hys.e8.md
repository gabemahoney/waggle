# Rename Waggle → Claude Spawn

## Problem Statement

The project is currently named "Waggle" — a bee-themed name inherited from the Apiary workflow. The product has stabilized as an orchestration surface for directing Claude Code workers; the name "Waggle" no longer signals what the tool does to anyone who hasn't been briefed. Renaming to "Claude Spawn" makes the project self-descriptive and aligns the surface name with the developer experience.

Several files in the tree describe and exercise removed daemon-era surface (`waggle serve`, `waggle set-state`, `waggle permission-request`, `waggle ask-relay`, the `waggle.service` systemd unit, the `~/.waggle/hooks/` path, the docker CI pipeline that drives them). They were stale before the rename was conceived. Renaming them mechanically would attach the new project name to obviously-broken artifacts; deleting them is the cheaper, more honest path while we're touching every file anyway.

## Goals

- Every project-owned identifier — repo dir, Python package, CLI binary, MCP server name, code symbols, env vars, label keys, tmux session prefix — uses "Claude Spawn" (or its derived forms).
- The codebase has zero references to "Waggle" after the refactor (excluding immutable Git history).
- Dead daemon-era files and code passages are removed, not renamed. The tree post-cutover reflects only the live stdio-MCP surface.
- The cutover is hard: no compatibility shims, no dual-name support.

## Non-Goals / Out of Scope

- Modifying Claude Status. Claude Status has no Waggle-specific knowledge in code; it stores opaque label key/value pairs. The rename only changes which keys Claude Spawn sets; Claude Status keeps reading them as it does today. If any Claude Status documentation mentions Waggle by name (as an example consumer), updating it is a follow-up coordinated with Claude Status's release cadence — not part of this PRD.
- Renaming bee tickets, Plan tickets, or Epic IDs in the bees database. Existing IDs (`b.m5a`, `b.fg8`, the seven Epic IDs) are immutable history. Commit messages on the rename branch may reference them.
- Coordinating a migration window for external operators. There are no external operators; only this development host runs Claude Spawn. The cutover is local and immediate.
- New functional surface — no new MCP tools, no new CLI subcommands, no behavioral tweaks to surviving code paths, no dependency upgrades, no performance work. The dead-code removal listed under **Dead-code removal** below is *subtractive only*: it deletes files and code passages that already target removed surface. No surviving feature changes.
- Renaming or re-aliasing the six MCP tool names (`spawn_worker`, `list_spawned_workers`, `terminate_worker`, `send_input`, `get_output`, `answer_question`). They are functional names, not project-branded, and orchestrators bind to them.
- Rebuilding a CI pipeline or installer for the new stdio-MCP surface. The docker/ image, `install.sh`, and the CI skills exist today only to drive the removed daemon. Replacements (if any) are a follow-up bee, not part of this rename.

## Requirements

### Functional Requirements

#### Identifiers

The repository directory name, the Python package directory, the CLI binary name, the MCP server's advertised `serverInfo.name`, and the `pyproject.toml` `name`/`description`/`packages`/`scripts` entries must all reflect the new name. The naming conventions are:

| Surface | Form | Value |
|---|---|---|
| Distribution name (`pyproject.toml [tool.poetry] name`) | kebab-case | `claude-spawn` |
| Python package (`src/<pkg>/`, `import <pkg>`) | snake_case | `claude_spawn` |
| CLI binary (`pyproject.toml [tool.poetry.scripts]`) | kebab-case | `claude-spawn` |
| MCP `serverInfo.name` (advertised by stdio server) | kebab-case | `claude-spawn-stdio` |
| Recommended operator-installed `mcpServers.<KEY>` (in `~/.claude.json` / `~/.claude/settings.json`) | kebab-case | `claude-spawn` |
| Display name in user-facing strings | Title Case | `Claude Spawn` |

#### Code symbols and imports

Every Python `import` statement that today reads `from waggle.X` or `import waggle.X` must be updated to the new package name. Every `__init__.py`, module docstring, and class docstring must use the new name. Every CLI help string, error message, and log message must use the new name. Every test-internal import and patch target (e.g. `patch("waggle.installer._run_install_hooks")`) must be updated.

#### Env vars Claude Spawn injects into worker tmux sessions

The env vars that Claude Spawn injects into worker tmux sessions must use a project-prefixed label namespace consistent with the rename. The chosen prefix is `CLAUDE_SPAWN_*`. The four label env vars are:

- `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_OWNED=1`
- `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_SESSION_NAME=<session_name>`
- `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_MODEL=<model>`
- `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_REPO=<repo>`

The resulting Claude Status `instances.labels` keys are `claude_spawn_owned`, `claude_spawn_session_name`, `claude_spawn_model`, `claude_spawn_repo`. The `_REQUIRED_ENV_VARS` tuple in the new `claude_spawn.spawn` module is updated accordingly.

The three non-label env vars Claude Spawn injects (`CLAUDE_STATUS_INSTANCE_ID`, `CLAUDE_STATUS_RELAY_MODE`, `CLAUDE_STATUS_AUQ_MODE`) are Claude-Status-owned and do not change.

#### Claude Status worker filter

The `list_spawned_workers` and `answer_question` implementations invoke `claude-status workers --label claude_spawn_owned=1` (replacing the previous `waggle_owned=1` filter). The `labels.waggle_session_name` lookup inside `answer_question_impl` becomes `labels.claude_spawn_session_name`.

#### Tmux session name prefix

The default tmux session name prefix changes from `waggle-<first8>` to `spawn-<first8>` — short enough to keep session names readable while still identifying ownership at a glance. (The mismatch with the longer `claude_spawn_*` label prefix is intentional: tmux session names are user-visible and brevity wins; labels are programmatic and explicitness wins.)

#### Sting MCP-detection pattern

`sting.py` detects whether the operator has registered Claude Spawn as an MCP server by regex-matching keys in the top-level `mcpServers` dict of `~/.claude.json` and `~/.claude/settings.json`. The current pattern `(?i)(?:^|[-_])waggle(?:$|[-_])` is replaced by:

```python
_CLAUDE_SPAWN_PATTERN = re.compile(r"(?i)(?:^|[-_])claude[-_]spawn(?:$|[-_])")
```

This pattern matches operator-chosen keys such as `claude-spawn`, `claude_spawn`, `my-claude-spawn`, `CLAUDE-SPAWN`. The Python symbol `_WAGGLE_PATTERN` is renamed to `_CLAUDE_SPAWN_PATTERN`; the helper `_key_matches_waggle` becomes `_key_matches_claude_spawn`; `_has_waggle_in_mcp_servers` becomes `_has_claude_spawn_in_mcp_servers`. All sting tests are updated to use the new symbol names and to exercise the new pattern's match cases.

#### Test files

Every test file whose name contains "waggle" (none today, but verify before the rename) must be renamed; every test-internal string referencing "Waggle" by name is updated. Tests that assert on label keys (e.g. `test_waggle_owned_label_1`) are renamed to match the new label keys (`test_claude_spawn_owned_label_1`).

#### Documentation

`README.md`, `docs/architecture/*.md`, and `docs/spikes/*.md` are rewritten to use the new name throughout.

`docs/migration.md` is rewritten without any literal occurrence of "Waggle". It describes the historical migration from the project's earlier HTTP-daemon distribution to the current stdio MCP distribution, in the voice of the new project name. Concrete consequences:

- Prose mentions of the prior project name are phrased obliquely (e.g. "this project was previously distributed under a different name").
- Historical filesystem paths that contained the old name as a literal substring — specifically `~/.waggle/` (daemon state dir) and `~/.config/systemd/user/waggle.service` (daemon systemd unit) — are not re-spelled with the new name (those paths never existed under the new name). They are described generically ("the daemon's state directory under `$HOME`", "the daemon's user-mode systemd unit"). Since the only operator has already migrated, the cleanup steps are historical narrative, not actionable instructions; oblique wording is acceptable.
- The "What changed" table row referencing removed daemon commands (`waggle serve`, `waggle set-state`, `waggle permission-request`, `waggle ask-relay`) is rephrased to describe the removed surface generically without naming the commands.

### Dead-code removal

The following files, directories, and code passages are deleted outright. They target removed daemon-era surface (`waggle serve`, `waggle set-state`, `waggle permission-request`, `waggle ask-relay`, the `waggle.service` systemd unit, `~/.waggle/hooks/`) and have no replacement value in the stdio-MCP era. Deletion is part of this PRD; rebuilding equivalents (if ever needed) is a follow-up bee.

| Path | Reason |
|---|---|
| `install.sh` | Installs a removed `waggle.service` systemd unit and writes hook entries for removed CLI commands. The current install path is `pip install . && claude-spawn install`. |
| `docker/Dockerfile` | Builds an image whose only purpose is to run the removed daemon flow. |
| `docker/entrypoint.sh` | Phase-dispatch entrypoint for the docker CI image; references `/tmp/waggle_phase_exit`. |
| `docker/phase1.sh` | Phase-1 CI script that validates `install.sh`. |
| `docker/phase2.sh` | Phase-2 CI script that starts `waggle serve` and exercises removed hook commands. |
| `docker/auto_approve.sh` | Auto-approval helper for the docker CI tmux flow; orphaned without docker/phase2.sh. |
| `docker/` (the directory itself, once empty) | No longer needed. |
| `plugins/opencode/state-tracker.ts` | Calls `~/.waggle/hooks/waggle_set_state.sh`, a path that no longer exists. |
| `plugins/opencode/` and `plugins/` (the directories themselves, once empty) | No longer needed. |
| `_CLI_REFERENCE` literal in `src/waggle/sting.py` (post-rename: `src/claude_spawn/sting.py`) | Describes removed `serve` and `set-state` subcommands. `handle_sting` does not print it; it has no live consumer outside its own test. |
| `TestCliReference` class in `tests/test_sting.py` | Sole consumer of `_CLI_REFERENCE`; removed alongside it. |
| `.claude/skills/ci/` (directory + `SKILL.md`) | Documents running the docker/ CI pipeline. Orphaned without docker/. |
| `.claude/skills/release-test/` (directory + `SKILL.md`) | Drives integration tests via the `tickets/testplans/` hive (already deleted from the working tree) and prints "WAGGLE CI PHASE 2 PASSED" — both removed surface. |

After the deletions, the import in `tests/test_sting.py` that pulls in `_CLI_REFERENCE` is removed, and any other test imports that reference deleted symbols are cleaned up so the suite collects without ImportError.

The `_remove_legacy_hook_template()` function in `claude_spawn.installer` (and its test in `tests/test_install_hooks.py`) stays: it cleans up a stale artifact on operator filesystems and is not daemon-era surface.

### Edge Cases and Error Handling

The rename + dead-code-removal is subtractive and identifier-rewriting. The runtime error contract for the surviving MCP tools (every tool returns a structured "operation failed" result on failure) is unchanged. No new error modes are introduced.

### Non-Functional Requirements

The renamed codebase must pass the test suite. Test logic is unchanged for surviving tests; the only test-side delta is the deletion of `TestCliReference` (1 test; net suite count drops from 213 to 212) and identifier/string renames inside the surviving 212 tests.

The renamed CLI must start within the same time budget as the current one (sub-second on a modern host with Claude Status installed).

- [ ] The Python package is importable as `import claude_spawn`; `import waggle` raises `ModuleNotFoundError`.
- [ ] The CLI is invoked as `claude-spawn` (kebab-case); `waggle` is not a recognized command.
- [ ] `claude-spawn --help` prints the new name in the program description.
- [ ] `claude-spawn mcp` launches an stdio MCP server whose `initialize` response advertises `serverInfo.name = "claude-spawn-stdio"`.
- [ ] `claude-spawn install` invokes `claude-status install-hooks` exactly as the previous `waggle install` did.
- [ ] `claude-spawn sting` reports the Claude Status install state and detects an `mcpServers` entry whose key matches `_CLAUDE_SPAWN_PATTERN`.
- [ ] A worker spawned via the new MCP server has `labels.claude_spawn_owned == "1"` in Claude Status.

- [ ] `poetry run pytest` passes with 212 tests collected and green.
- [ ] `README.md`, `docs/migration.md`, every file in `docs/architecture/`, and every file in `docs/spikes/` contain no occurrence of "Waggle" (case-insensitive).

- [ ] The following paths do not exist in the working tree: `install.sh`, `docker/`, `plugins/`, `.claude/skills/ci/`, `.claude/skills/release-test/`.

## Assumptions and Preconditions

- The dev host (`gmahoney-horde-controller`) is the only operator. No external installations need coordinated migration.
- **Pre-rename workers are terminated before cutover.** Any tmux worker carrying the legacy `waggle_owned=1` label is shut down before the rename ships; the new code's `claude_spawn_owned=1` filter will not see them, and there is no compatibility shim. (A worker that exists in Claude Status from before the rename remains queryable via raw `claude-status workers` but is invisible to `list_spawned_workers`.)
- The bees database tooling does not care about the rename; bee IDs and titles continue to work. The `bees` hive scope path (`/home/horde/projects/waggle-project/**`) is updated to `/home/horde/projects/claude-spawn-project/**` as part of the operator cutover steps.
- Git history is preserved (we rename in-tree; we do not rewrite history to retroactively rename old commits).
- The repository directory rename (`waggle-project` → `claude-spawn-project`) is done by a separate one-shot operator command (`mv` + worktree fix-up + bees scope update) and is not part of the code-level rename.
- The docker/ CI pipeline and `install.sh`-driven systemd installation flow have no surviving operator users; deleting them does not break any active workflow on this host.

## Decisions

- **Repo URL / Git remote — rename in place.** The existing repository becomes the new Claude Spawn repository. Operator steps for the cutover: `mv waggle-project claude-spawn-project` on the local filesystem, update `bees` scope to point at the new path, restart any persistent tmux sessions that cached the old cwd, and rename the GitHub remote (if any) using GitHub's built-in repo-rename feature so existing URLs auto-redirect. No new repo is created. Git history is preserved end-to-end.
- **No backwards-compat `waggle` CLI shim.** Hard cutover. After install, `waggle` is not a recognized command.
- **`pyproject.toml` distribution name uses kebab-case (`claude-spawn`)** to match the CLI binary; the Python package import name uses snake_case (`claude_spawn`) per Python convention.
- **Stale daemon-era surface is deleted, not renamed.** The files and code passages enumerated under **Dead-code removal** are removed as part of this PRD. A future CI rebuild (if needed) is out of scope.
