# Spawn Options and Templates for Claude Spawn Workers

## Problem Statement

Today's `spawn_worker` MCP tool exposes only three knobs (`model`, `repo`, `session_name`) and carries legacy "clone the URL" magic in `repo`, but operators already need to set more per-worker (thinking effort, Claude Code permissions, custom Claude config directory, instance ID, settings overlay, environment variables, status labels, extra CLI flags). Hand-passing those values on every spawn call is verbose, error-prone, and burns LLM tokens for orchestrators that spawn many similarly-shaped workers. There is no way today to save a worker shape and reuse it by name, and no way to know on a successful `spawn_worker` return that the worker is actually ready to receive input.

## Goals

- Operators can configure a richer set of per-worker options when invoking `spawn_worker`.
- Operators can define named "spawn templates" once and invoke them by name to avoid repeating option bundles.
- LLM orchestrators that spawn many similar workers can do so with a one-liner referencing a template.
- Templates can be authored programmatically via an MCP tool, interactively via a CLI walkthrough, or by hand-editing the underlying template file. All three paths produce identical on-disk templates.
- Callers can rely on a successful `spawn_worker` return: the worker is registered with Claude Status and ready to receive input before the call returns.
- The new surface is documented well enough that an operator can author a template and spawn from it without reading source code.

## Non-Goals / Out of Scope

- Template inheritance or `extends` chains. v1 templates are flat single-file definitions.
- Template field-level editing via MCP. `write_template` writes the whole file; partial patches and merges are not supported.
- Live reloading. Templates are read at spawn time; long-running orchestrator sessions pick up new template files on their next `spawn_worker` call.
- Validation of template contents against a formal JSON Schema. Best-effort parse with a clear error on malformed files is sufficient.
- Migration tooling from a prior spawn-options system. There is no prior system.
- Cloning Git URLs from `spawn_worker`. The old Waggle `repo` parameter that accepted HTTPS URLs and cloned them is dropped. Operators pre-clone before spawning, or instruct the spawned worker to clone via a follow-up prompt.
- An `initial_prompt` option that sends a tmux send-keys payload immediately after launch. Deferred — orchestrators can call `send_input` separately.
- Shipping any builtin templates with Claude Spawn. Operators author their own under `~/.claude-spawn/templates/`.

## Requirements

### Functional Requirements

#### Invocation shape

The orchestrator-facing tool signature, with all v1 options named:

```
spawn_worker(
    template?,
    model?, thinking?, cwd, tmux_session_name?, instance_id?,
    claude_home?, claude_settings?, extra_env?, claude_status_labels?,
    claude_args?, permissions?,
)
```

`cwd` is required either as a direct argument or supplied by the named `template`. Every other option is optional and falls back to its documented default.

#### Option resolution order

Each option's effective value is resolved as: explicit per-call argument → template field (if `template=<name>` is set) → the option's default per the fallback table below. Map-valued options (`extra_env`, `claude_status_labels`, `permissions`) merge across the chain by union, with later sources overridden by earlier ones on colliding keys.

#### Per-option fallback table

Every v1 option falls into one of three default categories.

**Required.** Caller (or template) must supply. Spawn fails without it.

| Option | Notes |
|--------|-------|
| `cwd` | Existing local filesystem directory; absolute or `~`-prefixed only. No default. |

**Hardcoded by Claude Spawn.** Claude Spawn picks a default when caller doesn't supply it.

| Option | Default |
|--------|---------|
| `tmux_session_name` | `<sanitized-folder-name>-<first 8 chars of instance_id>` |
| `instance_id` | A bare UUIDv4 (e.g. `c2157396-d7fb-471c-815c-fbe9db0f5ab4`) |
| `extra_env` | Empty map |
| `claude_status_labels` | Empty map |
| `claude_args` | Empty list |
| `permissions` | Empty map (no `allow`/`deny`/`ask` rules injected) |

**Inherits Claude Code's default.** Claude Spawn does not set; Claude Code's normal config-lookup chain (see below) applies.

| Option | Notes |
|--------|-------|
| `model` | No `--model` flag passed; Claude Code picks its own default. |
| `thinking` | No effort level injected; Claude Code uses its default. |
| `claude_home` | Worker inherits the spawn process's `HOME`, so Claude Code reads its normal `~/.claude/settings.json`. |
| `claude_settings` | No `--settings` overlay file; Claude Code reads only the layers below. |

#### Claude Code settings stack

`claude_home` and `claude_settings` compose with Claude Code's existing 4-layer settings lookup. Operators need this mental model to predict what their settings will resolve to.

The four layers Claude Code merges, from lowest to highest precedence:

1. **Enterprise managed settings** — a system-wide policy file (e.g. `/etc/claude-code/managed-settings.json`). Used by orgs to enforce policy.
2. **User settings** — `<HOME>/.claude/settings.json`. Per-user defaults.
3. **Project shared settings** — `<cwd>/.claude/settings.json`. Checked into the repo.
4. **Project local settings** — `<cwd>/.claude/settings.local.json`. Operator-local overrides; gitignored.

On top of that stack, the `--settings <path>` CLI flag injects an overlay file with the highest precedence: it wins over all four layers.

Claude Spawn's options map onto this stack:

- `claude_home` rewrites the worker's `HOME` env var, redirecting layer 2 (user settings) to a different config tree. Used to give a worker an isolated Claude config.
- `claude_settings` supplies the file passed to Claude Code via `--settings`. Used to inject a per-spawn overlay above all four layers.
- The first-class `permissions` option is realized by adding its `allow`/`deny`/`ask` keys into the same `--settings` overlay file Claude Spawn passes to Claude Code.

#### Spawn options surface

`spawn_worker` accepts the following options. `cwd` is required (directly or via a template); all others are optional and fall back to documented defaults when omitted.

- `model` — Claude model name (`opus`, `sonnet`, `haiku`). Default: Claude Code's default.
- `thinking` — Thinking-effort level, matching Claude Code's `/effort` levels (`low`, `medium`, `high`, `xhigh`). Default: Claude Code's default.
- `cwd` — The directory the Claude Code process starts in (its working directory on launch). Required. Must be an existing local filesystem directory, supplied as an absolute path or a `~`-prefixed path. Relative paths are rejected so the resolved location is never ambiguous about which process's working directory it is relative to. Claude Spawn does not clone Git URLs or fetch the code on the operator's behalf; pre-cloning is the operator's responsibility.
- `tmux_session_name` — The tmux session name the worker runs in. Visible to operators in `tmux list-sessions` and used by every tmux-keyed Claude Spawn tool (`send_input`, `get_output`, `terminate_worker`, `answer_question`). Default: `<sanitized-folder-name>-<first 8 characters of the resolved `instance_id`>`. The slice operates on whatever string `instance_id` resolves to — the auto-generated UUIDv4, a template-supplied value, or an operator-supplied value — without parsing it. If the resolved `instance_id` is shorter than 8 characters, the entire string is used (no padding). The sanitized folder name is produced by the folder-sanitize helper described in the Folder-name sanitize helper section below. Examples (with `cwd=/home/horde/projects/waggle-project/waggle-main` ⇒ folder `waggle-main`): default UUID `instance_id=c2157396-d7fb-471c-815c-fbe9db0f5ab4` ⇒ `waggle-main-c2157396`; custom `instance_id=my-custom-id` ⇒ `waggle-main-my-custo`; custom `instance_id=xyz` ⇒ `waggle-main-xyz`.
- `instance_id` — Override the auto-generated `CLAUDE_STATUS_INSTANCE_ID`. Default: a bare UUIDv4 (e.g. `c2157396-d7fb-471c-815c-fbe9db0f5ab4`); no folder prefix. The default `tmux_session_name` derives from this value (its first 8 characters), so the two identifiers share a visible prefix in `tmux list-sessions` — a human looking at `claude-status workers` and `tmux list-sessions` can pair a worker to its tmux session by matching the first 8 characters of the `instance_id` against the suffix of the `tmux_session_name`.
- `claude_home` — Override the worker process's `HOME` env var so Claude Code reads a different user-settings tree (`<claude_home>/.claude/settings.json`, i.e. layer 2 of the settings stack). Named explicitly `claude_home` (rather than just `home`) so it is clear what tree gets pointed at. Default: inherit the spawn process's `HOME`.
- `claude_settings` — Path to a JSON file passed to the worker's Claude Code process as `--settings`. This is a *raw* Claude Code `settings.json` overlay: it can contain anything Claude Code's settings schema accepts (e.g. `theme`, `defaultMode`, `mcpServers`, `enabledMcpjsonServers`, `disabledMcpjsonServers`, `customStatusLine`, `hooks`, `outputStyle`, `additionalDirectories`, `cleanupPeriodDays`, `apiKeyHelper`, `forceLoginMethod`). It is the power-user escape hatch for anything not surfaced as a first-class option. Default: not passed.
- `extra_env` — Map of additional environment variables (`{KEY: VALUE}`) injected into the worker's tmux session. Values are passed through without sanitization. Keys are open with one exception: keys that collide with Claude Spawn's built-in env-var set (`CLAUDE_STATUS_INSTANCE_ID`, `CLAUDE_STATUS_RELAY_MODE`, `CLAUDE_STATUS_AUQ_MODE`, any `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_*` key, or `HOME` when `claude_home` is also set) are rejected so callers cannot silently clobber Claude Spawn's control surface — see edge cases. Default: empty.
- `claude_status_labels` — Map of additional `CLAUDE_STATUS_LABEL_*` entries (`{KEY: VALUE}`) beyond Claude Spawn's built-in label set. Named explicitly so it is clear which system the labels land in. Default: empty.
- `claude_args` — List of extra command-line flags appended to the `claude` invocation (e.g. `--dangerously-skip-permissions`). The CLI surface for mode toggles that Claude Code does not expose through `settings.permissions`. Default: empty.
- `permissions` — A Claude Code permissions configuration applied at spawn time. Maps onto Claude Code's `settings.permissions` schema's tool-pattern lists: `allow`, `deny`, `ask`. Surfaced as a first-class option so passing a dict like `{allow: ["Bash", "Read(*)"], deny: ["WebFetch"]}` is ergonomic. Implementation folds these fields into the `claude_settings` overlay file passed via `--settings`. Mode toggles like `bypassPermissions` / `--dangerously-skip-permissions` are *not* part of this option; those go through `claude_args`. Default: empty.
- `template` — Name of a saved spawn template. When set, the template provides defaults for every other option; per-call options override on the keys they specify.

#### Folder-name sanitize helper

Used by the `tmux_session_name` default to produce the folder fragment that precedes the `instance_id` slice. Takes `cwd` (a filesystem path) and produces a non-empty, tmux-safe identifier:

1. `os.path.normpath(cwd)` — collapses trailing slashes and redundant separators.
2. `os.path.basename(normalized)` — the last path segment.
3. If the result is empty (e.g. `cwd="/"`) or equals `"."`, substitute literal `root`.
4. Replace every tmux-illegal character (`.`, `:`) with `-`.
5. If after replacement the result is still empty or all-punctuation, substitute literal `root`.

The output is always a non-empty string of `[A-Za-z0-9_-]+` characters, safe for use as the leading fragment of a tmux session name. Examples:
| `cwd` input | Sanitized folder |
|---|---|
| `/home/horde/projects/waggle-project/waggle-main` | `waggle-main` |
| `/home/horde/projects/waggle-project/waggle-main/` | `waggle-main` |
| `/path/foo.bar.tar.gz` | `foo-bar-tar-gz` |
| `/path/with:colon` | `with-colon` |
| `/` | `root` |
| `.` | `root` |

#### Spawn readiness

`spawn_worker` blocks until the worker has registered itself with Claude Status (i.e. is visible in the `instances` table). A successful return implies callers can immediately call `send_input`, `get_output`, or query Claude Status for that instance.

- The default readiness timeout is 15 seconds.
- If the worker has not registered within the timeout, `spawn_worker` returns an actionable readiness-timeout error, terminates the worker's tmux session to avoid leaking it, and the worker does not appear in the `instances` table.
- If the worker process exits before registering, `spawn_worker` returns an actionable error indicating the early exit and includes any captured tmux output for diagnostics.

#### Spawn templates

Templates are named bundles of spawn options.

- Stored as one file per template at `~/.claude-spawn/templates/<name>.<ext>`.
- Template name is the filename stem.
- Format is a human-readable, hand-editable text format. The PRD-author pick is TOML; the SRD finalizes the concrete format choice. All template-related requirements below assume the chosen format supports nested maps (needed for `extra_env`, `claude_status_labels`, and the `permissions` sub-dict).
- Loaded at spawn time when the caller passes `template=<name>`. There is no auto-loaded default template; a bare `spawn_worker(cwd=…)` call uses hardcoded option defaults, not any on-disk file.
- Merge semantics with per-call options: per-call options override template options key-by-key. For map fields (`extra_env`, `claude_status_labels`, `permissions`), the merge is a union, with per-call entries winning on key collisions.
- Every option settable per-call is also settable from a template. There is no template-only or per-call-only option. A template may supply the required `cwd`, removing the need for the caller to pass it explicitly.

#### Template discovery

A new MCP tool `list_templates` enumerates available templates.

- Returns one entry per file in `~/.claude-spawn/templates/`.
- Each entry includes the template name, its source file path, and its resolved option map.
- Malformed templates are excluded from the main result list and surfaced under a separate `skipped[]` channel with the file path and the parse error.

#### Template authoring (programmatic)

A new MCP tool `write_template(name, options, force?)` lets callers create or replace a template programmatically.

- `name` is the template's filename stem; Claude Spawn writes `~/.claude-spawn/templates/<name>.<ext>`. The name must be a single safe filesystem segment (no path separators, no leading dot, no `..`).
- `options` is a dict matching `spawn_worker`'s named parameters minus `template` itself (templates cannot recursively reference templates).
- `force` is an optional boolean defaulting to `false`. When a template by the same name already exists, the call fails unless `force=true` is passed.
- Unknown keys in `options`, unknown values (e.g. an unrecognized `thinking` level), or an unsafe `name` produce an actionable error and no file is written.
- On success the tool returns the resolved on-disk path and the option map that was serialized, so the caller can read back what was actually written.

#### Template authoring (CLI)

A new CLI subcommand `claude-spawn write-template <name>` provides two operator-facing modes that both delegate to `write_template`.

- **Interactive mode** — `claude-spawn write-template <name> --interactive` walks the operator through every spawn option one prompt at a time. Each prompt shows the option name, a short description, the current value (if a template with that name already exists), and accepts a value or the literal `skip`. After the last prompt the subcommand calls `write_template` with the collected options.
- **Flag-driven mode** — `claude-spawn write-template <name> --model=opus --thinking=xhigh --cwd=/path …` accepts each spawn option as a flag and forwards the bundle to `write_template`. Map-valued options (`extra_env`, `claude_status_labels`, `permissions`) accept repeatable `--key=value` flag forms documented in the README.
- Both modes accept `--force` to allow replacing an existing template; without it, a name collision is an error.
- Hand-editing template files under `~/.claude-spawn/templates/` continues to work. The authoring surfaces are convenience layers, not the only way in.

#### `spawn_worker` return shape

Returns `{instance_id, tmux_session_name}`. The previous return key `session_name` is renamed to `tmux_session_name` in lockstep with the parameter rename; no compatibility alias. Adding further fields would break the orchestrator-MCP contract.

- The Claude Code 4-layer settings stack and how `claude_home`, `claude_settings`, and `permissions` map onto it.
- The template file format with a worked example, and the merge rules.
- `list_templates` output shape.
- The `write_template` MCP tool and `claude-spawn write-template` CLI subcommand (both modes), including the `--force` flag and the repeatable-flag form for map options.

### Edge Cases and Error Handling

- **`cwd` missing entirely** — Neither per-call nor template supplies `cwd`. Spawn fails with an actionable error stating that `cwd` is required.
- **`cwd` is a path that does not exist** — Spawn fails with a clear error identifying the missing path. Claude Spawn does not create the directory or fall back.
- **`cwd` is an HTTPS URL or other non-path string** — Spawn fails with a clear error stating that `cwd` must be an existing local filesystem path and that URL cloning is the operator's responsibility.
- **`cwd` is a relative path (or any non-absolute, non-`~`-prefixed string)** — Spawn fails with the same path-format error as the URL case. Operators must pass an absolute or `~`-prefixed path so the resolved directory is not ambiguous about which process's working directory it is relative to.
- **`claude_settings` path does not exist or is not readable** — Spawn fails with a clear error identifying the path and the access problem. Claude Spawn does not create or substitute an empty file.
- **Operator-supplied `instance_id` collides with an active worker** — Spawn fails with an actionable error naming the colliding instance. Claude Spawn refuses to spawn rather than producing duplicate `instances` rows.
- **`claude_args` includes `--dangerously-skip-permissions` while `permissions` is also set** — Claude Code's bypass mode wins (it is a CLI-level toggle). The first-class `permissions` rules are effectively inert. This is operator intent; Claude Spawn does not reject the combination, but the README calls out the precedence.
- **`claude_args` includes `--settings <path>` while `claude_settings` is also set** — Spawn fails with an actionable error directing the operator to use the `claude_settings` option for the overlay instead of duplicating it via `claude_args`. Claude Spawn does not pass two `--settings` flags.
- **Unknown template name** — `spawn_worker(template="nope", …)` returns an actionable error naming the missing template and the directory searched. The worker is not spawned.
- **Malformed template file** — A template file that fails parsing or contains keys Claude Spawn does not recognize produces an actionable error from `spawn_worker(template=…)` identifying the file and the parse problem. The worker is not spawned. The same file appears under `list_templates`' `skipped[]` channel rather than its main result list.
- **Missing templates directory** — `~/.claude-spawn/templates/` does not exist. `list_templates` returns an empty result list and no error. `spawn_worker(template=…)` returns the unknown-template error.
- **Per-call options conflict with template options** — Per-call values win on the keys they specify. Map fields (`extra_env`, `claude_status_labels`, `permissions`) merge by union; per-call entries override template entries on colliding keys.
- **`extra_env` injects a value that breaks the worker** (e.g. `PATH=""`) — Operator's responsibility. Claude Spawn does not whitelist or sanitize env-var values; any value is passed through unchanged.
- **`extra_env` supplies a key reserved by Claude Spawn's built-in env set** — Spawn fails with an actionable error naming the offending key, before any tmux call is made. Reserved keys are the spawn-control env vars (`CLAUDE_STATUS_INSTANCE_ID`, `CLAUDE_STATUS_RELAY_MODE`, `CLAUDE_STATUS_AUQ_MODE`), every `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_*` label Claude Spawn emits, and `HOME` when `claude_home` is also set. Claude Spawn never lets a caller silently overwrite the built-in env set.
- **`permissions` and `claude_settings` both set permissions keys** — The first-class `permissions` option wins. Operators get the ergonomic surface as the source of truth; `claude_settings` cannot silently shadow it.
- **`thinking` set to a value Claude Code does not recognize** — Spawn fails with a clear error listing the accepted values (`low`, `medium`, `high`, `xhigh`). The same validation runs in `write_template`.
- **Worker fails to register within the readiness timeout** — `spawn_worker` returns a readiness-timeout error and tears down the worker's tmux session so it does not leak.
- **Worker process exits before registering** — `spawn_worker` returns an actionable error noting the early exit and includes captured tmux output for diagnostics.
- **`write_template` with a name that collides with an existing template** — Fails with an actionable error naming the existing file unless `force=true` is set. No file is written or modified on failure.
- **`write_template` with an unsafe `name`** — A name containing path separators, a leading `.`, `..`, or characters that would escape `~/.claude-spawn/templates/` is rejected with an actionable error and no file is written.
- **`write_template` with unknown option keys** — The call fails listing the unrecognized keys and the full set of valid keys. No partial template is written.
- **`write_template` with an `options` dict containing `template`** — Rejected; templates cannot reference templates. No file is written.
- **`claude-spawn write-template --interactive` cancelled mid-walkthrough** — No file is written. The subcommand exits cleanly with a non-zero status and a message indicating cancellation.

### Non-Functional Requirements

- Template parsing is fast enough that `list_templates` runs in well under a second even with dozens of templates present.
- The spawn-readiness wait does not regress the happy-path spawn latency by more than the time the worker actually takes to register; the timeout is an upper bound, not a fixed wait.
- Tests cover: parameter handling for every new spawn option; template merge semantics including map-field union; the `permissions`-vs-`claude_settings` conflict resolution; the required-`cwd` paths; the malformed-template path through both `spawn_worker` and `list_templates`; the `write_template` write path (success, collision-without-`force`, collision-with-`force`, unsafe `name`, unknown keys, and `template`-in-options rejection); and the spawn-readiness paths (timely registration, registration-timeout cleanup, worker-exits-early).

## Acceptance Criteria

- [ ] A worker spawned via `spawn_worker(template="<name>", …)` has its `claude_status_labels` entries visible under the worker's Claude Status `labels` field, with per-call overrides applied on top of template values.
- [ ] A worker spawned with non-label `extra_env` entries has those variables present in the worker process's environment (verifiable by inspecting the worker's env or by having the worker echo them), even though they do not appear in Claude Status's `labels`.
- [ ] A worker spawned with `thinking="xhigh"` (per-call or from template) launches with that effort level applied; the operator can confirm via Claude Code's effort indicator.
- [ ] A worker spawned with `permissions={allow:["Bash"], deny:["WebFetch"]}` (per-call or from template) has those rules in effect at first prompt — `Bash` runs without prompting, `WebFetch` is denied.
- [ ] When `permissions` and `claude_settings` both supply permissions keys, the `permissions` option wins.
- [ ] A worker spawned with `cwd=<some-local-dir>` starts its Claude Code process in that directory; Claude Status's `cwd` field reflects it.
- [ ] `spawn_worker()` with no `cwd` (neither per-call nor template) returns an actionable error and does not spawn a worker.
- [ ] `spawn_worker(cwd="<nonexistent-path>")` returns an actionable error and does not spawn a worker.
- [ ] `spawn_worker(cwd="https://github.com/foo/bar")` (or any other non-path string) returns an actionable error indicating URL cloning is the operator's responsibility and does not spawn a worker.
- [ ] `spawn_worker(cwd="./some-relative")` (or any other relative, non-`~`-prefixed path) returns an actionable error and does not spawn a worker, even if the resolved path exists.
- [ ] `spawn_worker(claude_settings="<nonexistent-path>")` returns an actionable error identifying the missing path and does not spawn a worker.
- [ ] `spawn_worker(instance_id="<existing>")` where `<existing>` is the instance_id of an active worker returns an actionable error and does not spawn a worker.
- [ ] `spawn_worker(claude_args=["--settings","/x"], claude_settings="/y")` returns an actionable error rather than passing two `--settings` flags to Claude Code.
- [ ] `spawn_worker(extra_env={"CLAUDE_STATUS_INSTANCE_ID": "x"})` (or any other reserved built-in env-var key per the edge-cases list) returns an actionable error naming the offending key and does not spawn a worker.
- [ ] Per-call options override template options on the keys they specify; map fields (`extra_env`, `claude_status_labels`, `permissions`) merge by union with per-call wins on collisions.
- [ ] A template that supplies `cwd` satisfies the required-`cwd` rule; the caller need not pass `cwd` explicitly.
- [ ] `list_templates` returns one entry per file in `~/.claude-spawn/templates/`, each with name, source file path, and resolved option map.
- [ ] A malformed template file produces an actionable error from `spawn_worker(template=…)` and is surfaced under `list_templates`' `skipped[]` channel.
- [ ] `spawn_worker(template="<missing>", …)` returns an actionable error naming the missing template and does not spawn a worker.
- [ ] A bare `spawn_worker(cwd=…, …)` call (no `template=…`) consults no on-disk template.
- [ ] `spawn_worker(…)` returns only after the worker is visible in Claude Status's `instances` table; an immediate `send_input` against the returned `tmux_session_name` reaches the worker.
- [ ] If the worker fails to register within the readiness timeout, `spawn_worker` returns an actionable readiness-timeout error and the worker's tmux session has been torn down (no orphaned session in `tmux list-sessions`).
- [ ] If the worker process exits before registering, `spawn_worker` returns an actionable error including captured tmux output.
- [ ] `write_template(name="foo", options={...})` writes the template file under `~/.claude-spawn/templates/`. The resulting file is loadable by `spawn_worker(template="foo", …)` and produces a worker reflecting the supplied options.
- [ ] `write_template(name="foo", options={...})` against a name that already exists returns an actionable error and does not modify the existing file. Calling again with `force=true` succeeds and replaces the file.
- [ ] `write_template` with an unsafe `name`, with unknown option keys, or with `template` in `options` returns an actionable error and writes no file.
- [ ] `claude-spawn write-template <name> --interactive` walks every v1 spawn option, accepts a value or `skip` for each, and on completion produces a template file equivalent to a `write_template` call with the same option bundle.
- [ ] `claude-spawn write-template <name> --model=opus --thinking=xhigh --cwd=/path …` produces a template file equivalent to a `write_template` call with the same options. Repeatable flags for `extra_env`, `claude_status_labels`, and `permissions` are documented in the README and work as documented.
- [ ] Both CLI modes honor `--force`; without it, name collisions fail with the same error surface as the MCP tool.

## Assumptions

- Claude Spawn runs in a single-operator-host context where `extra_env` is safe to leave open (any KEY=VALUE allowed). Operator footguns are the operator's responsibility.
- Claude Code's `--settings` overlay merges additively on top of its 4-layer settings stack (enterprise → user → project shared → project local). The new `claude_home` option redirects layer 2; `claude_settings` injects the `--settings` overlay; the first-class `permissions` option is realized inside that overlay.
- Claude Code's `/effort` levels (`low`, `medium`, `high`, `xhigh`) can be set non-interactively at launch — either via a CLI flag, a `settings.json` field, or an environment variable. Implementation picks the mechanism; the PRD only requires that the spawned worker comes up with the requested effort level applied.
- Claude Code's `settings.permissions` schema's `allow` / `deny` / `ask` tool-pattern lists are the canonical surface for per-tool permissions; mode toggles like `bypassPermissions` are not part of the first-class `permissions` option and are reached via `claude_args` instead.
- Claude Status's `instances` table is the canonical readiness signal: a worker that appears there is considered ready to receive input.
- **Hard dependency on the Waggle → Claude Spawn rename.** This work assumes the rename Plan (Idea Bee `b.hys`, Plan Bee `b.fkb`) has shipped first. Every "Claude Spawn" reference in this PRD — the `claude-spawn` CLI binary, the `~/.claude-spawn/templates/` config directory, the `CLAUDE_STATUS_LABEL_CLAUDE_SPAWN_*` label namespace, the `claude_spawn` Python package — presumes the rename is complete. `b.dqt` cannot land until the rename has merged; until then, the names in this PRD will not match the codebase.

### PRD-author decisions

The following lockings resolve open questions enumerated in earlier revisions of the Idea Bee. They are the PRD's authoritative answers and stand unless the SRD or PRD review surfaces evidence to overturn one.

- **v1 option list** — The twelve options listed in the invocation shape ship in v1: `template`, `model`, `thinking`, `cwd`, `tmux_session_name`, `instance_id`, `claude_home`, `claude_settings`, `extra_env`, `claude_status_labels`, `claude_args`, `permissions`. `initial_prompt` is deferred as thin convenience over `send_input`.
- **`extra_env` policy** — Open dict, no whitelist or sanitization.
- **Default-template auto-apply** — Explicit-only. A bare `spawn_worker()` call uses hardcoded defaults; only `spawn_worker(template=…)` reads from disk.
- **Per-call vs. template-only asymmetry** — Every option is overridable per-call. No template-only or per-call-only options.
- **Builtin templates** — None shipped. README documents an authoring example; operators author their own under `~/.claude-spawn/templates/`. Sidesteps the install-path question.
- **`permissions` vs `claude_settings` conflict** — First-class `permissions` wins. Keeps the ergonomic surface authoritative.
- **`write_template` overwrite safety** — Refuse by default; opt-in to overwrite via `force=true` (MCP) / `--force` (CLI). Same behavior in both surfaces.

## Open Questions

- **Template file format.** PRD-author pick is TOML, on the strength of matching Claude Status's config-file format and supporting nested maps natively. The Idea Bee deferred the concrete format choice to the SRD; the SRD should confirm TOML or pick an alternative human-readable, hand-editable text format that supports nested maps.
- **Spawn-readiness timeout default.** PRD-author pick is 15 seconds, within the Idea Bee's 10–30 second range. The right value depends on observed worker startup times in practice; the SRD or operational feedback may tune it. The PRD assumes the value is a hardcoded constant in v1 — no per-call or template-level override — until evidence suggests operators need to tune it.
