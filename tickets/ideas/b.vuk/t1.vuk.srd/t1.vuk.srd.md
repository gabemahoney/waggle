## SR-2: install.sh

### SR-2.1: Location and Permissions
- `install.sh` lives at the project root (same directory as `pyproject.toml`)
- Must be marked executable

### SR-2.2: Repo Detection
- Before any other operations, verify `pyproject.toml` exists in the same directory as the script
- On failure: print a clear error message and exit non-zero

### SR-2.3: Idempotency Gate
- Check `claude mcp list` output; if a server named `waggle` appears, print "Waggle is already installed." and exit 0 without modifying any system state
- This check gates the entire install path — nothing runs if waggle is already registered

### SR-2.4: Python Dependencies
- Run `poetry install` to install all dependencies defined in `pyproject.toml`
- If `poetry` is not found in PATH, print instructions and exit non-zero

### SR-2.5: MCP Registration
- Register waggle using:
  ```
  claude mcp add --transport stdio --scope user waggle -- poetry run --directory <absolute-path> waggle
  ```
  where `<absolute-path>` is the absolute path to the project root, resolved dynamically at script runtime
- If `claude` is not found in PATH, print instructions and exit non-zero

### SR-2.6: Hook Script Deployment
- Copy `hooks/waggle_set_state.sh` from the project root to `~/.waggle/hooks/waggle_set_state.sh`
- Create `~/.waggle/hooks/` if it does not exist
- Set the deployed script executable

### SR-2.7: Hooks Merge Into settings.json
- Target file: `~/.claude/settings.json`
- If file does not exist: create it with the full hooks configuration block
- If file exists: scan all hook `command` values across all events; if any contains `waggle_set_state.sh`, skip the merge and print a notice that hooks are already configured
- The merged configuration must contain all events and matchers defined in the README Claude Code Setup section, with `waggle_set_state.sh` as the hook script name throughout

### SR-2.8: Uninstall
- `--uninstall` invokes `claude mcp remove waggle`
- `--uninstall` removes all entries from `~/.claude/settings.json` whose `command` value contains `waggle_set_state.sh`; if an event array becomes empty after removal, that event key is also removed
- `--uninstall` does not remove `~/.waggle/` or any files within it
- `--uninstall` prints confirmation of each action taken

### SR-2.9: Output
- On successful install: print each action taken (dependencies installed, MCP registered, hooks deployed, settings.json updated)
- On any error: print a clear description and exit non-zero

## SR-4: Docker Image

### SR-4.1: Base and Python Version
- Base image: Ubuntu 24.04 LTS
- Python version: 3.12

### SR-4.2: Required Software
- Poetry (latest stable, installed via the official installer)
- tmux (from Ubuntu 24.04 apt)
- Node.js LTS (via official NodeSource apt repository)
- Claude CLI: installed globally via `npm install -g @anthropic-ai/claude-code`
- pipx (installed via pip into the system Python)
- bees CLI with MCP server: installed via `pipx install 'bees-md[serve]'`

### SR-4.3: Non-Root User
- A non-root user named `waggle` with UID 1000 must be created in the image
- Home directory: `/home/waggle`
- All Claude CLI state, waggle install, and poetry virtualenvs must reside under `/home/waggle`
- The container default user must be `waggle` (Dockerfile `USER` directive)

### SR-4.4: Source Inclusion
- Waggle source is available at `/opt/waggle` inside the container, owned by the `waggle` user
- Source is baked in via `COPY` from a staged build context — not a runtime mount
- rsync exclusion list before staging: `.git`, `.venv`, `__pycache__`, `*.pyc`, `.pytest_cache`, `tickets`, `bugs`

### SR-4.5: Build Policy
- Image is always built from source; no pre-built registry images
- The `/ci` skill always builds with flags that ensure the image reflects current source

## SR-5: /ci Skill

### SR-5.1: Location
- File: `.claude/skills/ci/SKILL.md`

### SR-5.2: Pre-flight Checks
- Check Docker is running via `docker info`; on failure print "Start Docker Desktop and retry." and exit
- Retrieve the Claude API key from the macOS Keychain following the exact approach in `/Users/gmahoney/projects/bees_project/bees_main/docker/run_test.sh`:
  1. Try `security find-generic-password -s "Claude Code" -w`
  2. If empty, try `security find-generic-password -s "Claude Code-credentials" -w`, parse the resulting JSON, and extract `.claudeAiOauth.accessToken`
  3. If still empty: print "No Claude Code credentials in Keychain. Run 'claude /login' first." and exit

### SR-5.3: Pre-run Cleanup
- Remove any existing containers named `waggle-ci-1` and `waggle-ci-2`
- Kill any existing tmux sessions named `waggle-ci-1` and `waggle-ci-2`

### SR-5.4: Source Staging
- rsync waggle source to a temp directory applying the exclusion list from SR-4.4
- The temp directory serves as the Docker build context

### SR-5.5: Phase Execution
- Each phase runs in a Docker container named `waggle-ci-<N>`
- Inside the container, the test runner executes in a tmux session named `ci`
- On the host, a tmux session named `waggle-ci-<N>` is created that wraps:
  `docker exec -it -u waggle waggle-ci-<N> tmux attach -t ci`
  This allows the developer to observe any phase with `tmux attach -t waggle-ci-<N>` — following the pattern in `/Users/gmahoney/projects/bees_project/bees_main/docker/run_test.sh`
- The `/ci` skill prints "Attach with: tmux attach -t waggle-ci-<N>" after starting each phase
- Phase 2 runs only if Phase 1 exits 0
- Both phases receive `ANTHROPIC_API_KEY` as a container environment variable
- The Phase 2 container mounts `<project-root>/tickets/testplans/` from the host as a read-only volume at `/tmp/testplans_host` inside the container — following the pattern in `/Users/gmahoney/projects/bees_project/bees_main/docker/run_test.sh`

### SR-5.6: Failure Handling
- On phase failure: print the last 30 lines of `docker logs <container>` and halt
- No retries or recovery attempts

### SR-5.7: Completion Cleanup
- After all phases pass or on any failure: remove created containers and kill created tmux sessions
- Print "All phases passed." on full success

## SR-6: Phase 1 — Install Verification

### SR-6.1: Install Execution
- Run `/opt/waggle/install.sh` as the `waggle` user inside the container
- Phase 1 fails if `install.sh` exits non-zero

### SR-6.2: MCP Registration Check
- After `install.sh`, verify `claude mcp list` output contains `waggle`

### SR-6.3: Entrypoint Verification
- `waggle.server:run` starts an MCP server loop and accepts no CLI arguments — it must NOT be invoked directly
- Verification must confirm the module is importable without starting the server: run `poetry run python -c "import waggle.server"` from `/opt/waggle` and verify it exits 0
- Additionally verify the `waggle` script exists in the poetry virtualenv bin directory

### SR-6.4: Success Signal
- Print `WAGGLE CI PHASE 1 PASSED` on success

## SR-7: Testplans Hive

### SR-7.1: Hive
- A bees hive named `testplans` is created at `tickets/testplans/` in the waggle repo and checked into source control
- Inside the Phase 2 container, the hive is available at `/tmp/testplans_host` (mounted as a read-only volume per SR-5.5)
- The Phase 2 container setup writes a bees config file at `/tmp/testplans-config.json` pointing only to `/tmp/testplans_host` — isolated from any host bees config
- The bees MCP server is registered with Claude CLI inside the container using:
  ```
  claude mcp add bees -- bees serve --stdio --config /tmp/testplans-config.json
  ```
  following the pattern in `/Users/gmahoney/projects/bees_project/bees_main/docker/test_runner.sh`

### SR-7.2: Test Tickets
Six bee tickets, one per MCP tool. All tickets are self-contained: each performs its own setup and teardown. No `up_deps` between tickets.

1. **list_agents** — call with no active sessions; verify the response is a valid (possibly empty) agent list
2. **spawn_agent** — spawn a Claude agent in `/tmp/waggle-test-spawn`; verify `status: success` and session appears in `list_agents`; teardown: close the session via `close_session`
3. **read_pane** — spawn a Claude agent in `/tmp/waggle-test-read`; read its pane; verify `agent_state` and `content` are present; teardown: close the session
4. **send_command** — spawn a Claude agent in `/tmp/waggle-test-send`; wait for `done` state; send a benign command; verify `status: success`; teardown: close the session
5. **close_session** — spawn a Claude agent in `/tmp/waggle-test-close`; close it; verify it no longer appears in `list_agents`
6. **delete_repo_agents** — insert a stale DB row directly into `~/.waggle/agent_state.db` via `sqlite3` for a test repo path; invoke `delete_repo_agents` for that path; verify `deleted_count >= 1`; teardown: remove any remaining rows for the test repo path

### SR-7.3: Ticket Content Requirements
- Each ticket body must specify: input parameters, expected response fields, and pass/fail criteria precise enough that no assumptions are needed during execution

## SR-8: release-test Skill

### SR-8.1: Location
- File: `.claude/skills/release-test/SKILL.md`

### SR-8.2: Invocation
- Claude is launched inside the Phase 2 container tmux session as:
  ```
  exec claude "/release-test"
  ```
  following the pattern in `/Users/gmahoney/projects/bees_project/bees_main/docker/test_runner.sh`
- No `--dangerously-skip-permissions` flag is used
- Permission prompts are handled by an `auto_approve.sh`-style script running in the background, monitoring the tmux pane and sending approval keystrokes — following the implementation at `/Users/gmahoney/projects/bees_project/bees_main/auto_approve.sh`

### SR-8.3: Test Execution
- Fetches all tickets from the local testplans hive via bees MCP tools
- Executes tickets in the order returned by the bees query (no dependency ordering required — tickets have no `up_deps`)
- Prints `[N/TOTAL] PASS: <title>` or `[N/TOTAL] FAIL: <title> — <error>` after each test

### SR-8.4: Failure Behavior
- On first failure: print the FAIL line and stop — no further tests run, no fixes attempted
- The skill must exit such that the Phase 2 container does not print `WAGGLE CI PHASE 2 PASSED`, which the `/ci` skill treats as failure

### SR-8.5: Success Signal
- After all tests pass: print `WAGGLE CI PHASE 2 PASSED`
