# Waggle Install Script and CI Pipeline

## Problem Statement

Installing waggle requires users to manually edit `~/.claude.json` and `~/.claude/settings.json` — a non-standard, error-prone process. There is also no automated way to verify that a fresh install works or that the MCP server's tools function correctly after a code change.

## Goals

- A single `install.sh` script handles full waggle setup: dependencies, MCP registration, and hooks.
- The README's setup section directs users to `install.sh` with an Advanced Usage section documenting what it does internally.
- A `/ci` skill launches a Docker-based test pipeline that validates both install and MCP tool functionality.
- After CI runs, the developer knows whether waggle installs and works correctly in a clean environment.

## Non-Goals / Out of Scope

- OpenCode setup changes (that section stays as-is).
- Publishing waggle to PyPI or any package index.
- CI for any host other than macOS with Docker Desktop.
- Automated triggering of CI (e.g. on git push) — this is a manual developer tool only.
- Pinning the Python version in the Dockerfile (specified in SRD — must exist before implementation).

## Requirements

### Functional Requirements

**install.sh**

The script lives at the project root. It must:

- Check `claude mcp list` first. If waggle is already registered, print "Waggle is already installed." and exit 0.
- Run `poetry install` to install Python dependencies.
- Register the waggle MCP server using the following command (path resolved dynamically from the script's location at run time):
  ```bash
  claude mcp add --transport stdio --scope user waggle -- poetry run --directory <absolute-path> waggle
  ```
  All options must precede the server name; `--` separates the name from the command. Reference: [Claude Code MCP docs](https://code.claude.com/docs/en/mcp).
- Merge the waggle hooks configuration into `~/.claude/settings.json`. If the file does not exist, create it. Detection: if any existing hook command contains `waggle_set_state.sh`, waggle hooks are considered installed — skip the merge entirely.
- Print clear success output when complete.

The script must accept an `--uninstall` flag that:
- Removes the waggle MCP server entry via `claude mcp remove waggle`.
- Removes all hook entries from `~/.claude/settings.json` whose command string contains `waggle_set_state.sh`.
- Prints confirmation when complete.

**README update**

- The Quick Start section must direct users to run `install.sh` as the setup step for Claude Code.
- All existing manual JSON blocks for `~/.claude.json` and `~/.claude/settings.json` must be moved to a new "Advanced Usage" section that documents what `install.sh` does internally.
- The "Verify Setup" step (`claude mcp list`) must be retained.
- The `--uninstall` flag must be documented.

**`/ci` skill**

A skill at `.claude/skills/ci/SKILL.md` that orchestrates a Docker-based test run. The skill must:
- Check Docker is running before proceeding.
- rsync the waggle source to a temp build context (excluding `.git`, `.venv`, `__pycache__`) and bake it into the Docker image via `COPY`.
- The waggle source is available inside the container at `/opt/waggle`.
- Build a Docker image containing: Ubuntu, Python (version per SRD), Poetry, tmux, Node.js, and the Claude CLI.
- Run two sequential test phases, each in its own container and tmux session named `waggle-ci-<N>`.
- Pull the Claude API key from the macOS Keychain. The SRD implementation should follow the same approach used in the bees repo (`/Users/gmahoney/projects/bees_project/`) — reference that CI implementation for the exact `security` command and keychain service name.
- After all phases pass, report "All phases passed."
- On any failure, print the last 30 lines of container output and stop.
- Clean up containers and tmux sessions from previous runs before starting.

**Phase 1 — Install**

Runs first and must pass before Phase 2 starts. Inside the container:
- Run `/opt/waggle/install.sh` to set up waggle.
- Verify `claude mcp list` output includes a `waggle` entry.
- Verify the waggle module is importable (`poetry run python -c "import waggle.server"` exits 0) and the `waggle` script exists in the poetry virtualenv bin directory.

Print `WAGGLE CI PHASE 1 PASSED` on success.

**Phase 2 — Functionality**

Runs after Phase 1 passes. Inside the container:
- A testplans hive is pre-populated with one self-contained test per MCP tool.
- A Claude instance is launched with the `release-test` skill to execute the tests.
- Tests cover each tool at least once: `list_agents`, `spawn_agent`, `read_pane`, `send_command`, `close_session`, `delete_repo_agents`.
- The actual test cases are defined in the SRD.

Print `WAGGLE CI PHASE 2 PASSED` on success.

**`release-test` skill**

A skill at `.claude/skills/release-test/SKILL.md` that runs inside the container. It must:
- Fetch the test plan from the local testplans hive.
- Execute each test in dependency order.
- Print `[N/TOTAL] PASS: <name>` or `[N/TOTAL] FAIL: <name> — <error>` after each test.
- On failure: print the FAIL line and exit immediately without attempting fixes.

### Edge Cases and Error Handling

- If `install.sh` is run from outside the waggle repo, it must detect this (e.g. absence of `pyproject.toml`) and exit with a clear error.
- If `claude mcp list` shows waggle already registered, `install.sh` prints "Waggle is already installed." and exits 0 without making any changes.
- If `~/.claude/settings.json` already contains any hook command with `waggle_set_state.sh`, `install.sh` must not add duplicates.
- If Docker is not running when `/ci` is invoked, the skill must print instructions to start Docker and exit cleanly.
- If no Claude credentials are found in Keychain, the skill must print "No Claude Code credentials in Keychain. Run 'claude /login' first." and exit.
- If Phase 1 fails, Phase 2 must not start.
- If a container crashes unexpectedly, the skill grabs the last 30 lines of `docker logs` and reports them.

### Non-Functional Requirements

- The Docker image is built from source (not pulled from a registry).
- The container runs as a non-root user to simulate a realistic user environment.
- CI must clean up containers and tmux sessions from previous runs before starting.

## Acceptance Criteria

- [ ] Running `install.sh` from a fresh clone completes setup with no manual JSON editing.
- [ ] `claude mcp list` shows waggle registered after `install.sh`.
- [ ] `~/.claude/settings.json` contains the waggle hooks after `install.sh`.
- [ ] Running `install.sh` a second time prints "Waggle is already installed." and exits cleanly.
- [ ] `install.sh --uninstall` removes both the MCP entry and hook entries.
- [ ] README Quick Start section shows `install.sh` as the setup step.
- [ ] README Advanced Usage section documents what `install.sh` does internally.
- [ ] `/ci` fails fast with a clear message if Docker is not running.
- [ ] `/ci` Phase 1 verifies install succeeds in a clean Docker environment.
- [ ] `/ci` Phase 2 exercises all 6 MCP tools and passes.
- [ ] Phase 2 does not start if Phase 1 fails.
- [ ] Each phase is observable via `tmux attach -t waggle-ci-<N>`.
- [ ] All containers and tmux sessions are cleaned up after CI completes.

## Assumptions

- waggle is always run from source via `poetry run` — no PyPI release planned.
- The developer running `/ci` has Docker Desktop installed and Claude CLI authenticated.
- The Claude API key retrieval method mirrors the bees repo CI implementation (`/Users/gmahoney/projects/bees_project/`).
- The waggle MCP server can be meaningfully tested inside a Docker container that has tmux available.
- Python version for the Dockerfile is 3.12 (specified in SRD).
