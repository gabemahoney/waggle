## Context
The `read_pane` tool (Epic t1.kN24) will need to classify raw pane text into agent states. This parser is a pure-function module with no tmux dependency and no I/O — it operates on plain strings only (SR-3.1). It returns an agent state classification and optional structured prompt data.

## What Needs to Change
- Create `src/waggle/state_parser.py` with:
  - A main function that accepts raw pane text and returns (agent_state, prompt_data)
  - Working detection (SR-3.2): detect `Esc to interrupt` indicator
  - Done detection (SR-3.3): detect empty `>` prompt at bottom, not `>` in output text
  - AskUserQuestion detection (SR-3.4): detect `❯` selector + `───` separator, extract structured prompt_data (question text, options list)
  - CheckPermission detection (SR-3.5): detect `Do you want to proceed?` + `Permission rule`, extract prompt_data (tool type, command, description)
  - Unknown fallback for unrecognized content
  - Priority ordering (SR-3.6): CheckPermission > AskUserQuestion > Working > Done > Unknown

## Why
Every downstream interaction tool needs to know what state an agent is in before it can act.

## Success Criteria
- Module imports with no side effects and no external dependencies beyond Python stdlib
- Parse function returns correct state for each of the 5 states
- prompt_data populated with structured data for ask_user and check_permission
- prompt_data is None for working, done, unknown
- Priority ordering prevents misclassification when patterns overlap
